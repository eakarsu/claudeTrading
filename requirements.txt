# Advanced Backtesting Analysis

## Analyze the buy/entry and sell/exit tags

Understanding how a strategy performs with different buy/entry tags helps identify which conditions are most effective. The backtesting-analysis command provides detailed statistics beyond standard backtesting output, including indicator values at signal candles.

This analysis is restricted to backtesting operations and is unavailable in hyperopt mode.

### Initial Setup

Run backtesting with the signals export option enabled:

`freqtrade backtesting -c <config.json> --timeframe <tf> --strategy <strategy_name> --timerange=<timerange> --export=signals`

This exports a pickled dictionary containing strategy data, pairs, and DataFrames of candles that triggered entry and exit signals. Be aware that files may grow large depending on entry frequency, so monitor your `user_data/backtest_results` directory regularly.

Before subsequent backtests, either remove old results or use `--cache none` to avoid cached data.

Upon successful completion, two files appear in `user_data/backtest_results`:
- `backtest-result-{timestamp}_signals.pkl`
- `backtest-result-{timestamp}_exited.pkl`

### Running the Analysis

Execute the analysis command with grouping options:

`freqtrade backtesting-analysis -c <config.json> --analysis-groups 0 1 2 3 4 5`

The `--analysis-groups` parameter specifies tabular output detail levels:

- **0:** Overall winrate and profit summary by enter tag
- **1:** Profit summaries grouped by enter tag
- **2:** Profit summaries grouped by enter tag and exit tag
- **3:** Profit summaries grouped by pair and enter tag
- **4:** Profit summaries grouped by pair, enter tag, and exit tag (can be large)
- **5:** Profit summaries grouped by exit tag

Additional options are available via the `-h` flag.

### Using backtest-filename

By default, the tool processes the most recent backtest results. To analyze earlier results, specify the desired file:

`freqtrade backtesting-analysis -c <config.json> --backtest-filename backtest-result-2025-03-05_20-38-34.zip`

For results in different directories, use `--backtest-directory`:

`freqtrade backtesting-analysis -c <config.json> --backtest-directory custom_results/ --backtest-filename backtest-result-2025-03-05_20-38-34.zip`

### Tuning the buy tags and sell tags to display

Filter displayed tags using these options:

- `--enter-reason-list`: Space-separated entry signals to analyze (default: "all")
- `--exit-reason-list`: Space-separated exit signals to analyze (default: "all")

Example filtering specific tags:

`freqtrade backtesting-analysis -c <config.json> --analysis-groups 0 2 --enter-reason-list enter_tag_a enter_tag_b --exit-reason-list roi custom_exit_tag_a stop_loss`

### Outputting signal candle indicators

Display indicator values at signal candles using:

`freqtrade backtesting-analysis -c <config.json> --indicator-list rsi rsi_1h bb_lowerband ema_9 macd macdsignal`

Indicators must exist in your strategy's main DataFrame for the specified timeframe, otherwise they are ignored.

#### Accessible Fields

Trade and candle-related fields automatically available include:

- **Datetime:** open_date, close_date
- **Price Levels:** min_rate, max_rate, open, close, high, low
- **Volume and Profitability:** volume, profit_ratio, profit_abs

#### Sample Output for Indicator Values

Example command displaying Ichimoku indicators:

`freqtrade backtesting-analysis -c user_data/config.json --analysis-groups 0 --indicator-list chikou_span tenkan_sen`

Output shows indicator values with `(entry)` and `(exit)` suffixes denoting timing.

The `(entry)` suffix reflects values at trade entry; `(exit)` reflects values at trade exit.

#### Filtering Indicators Based on Entry or Exit Signals

Restrict indicator display to specific signal types:

- `--entry-only`: Display indicator values exclusively at entry signals
- `--exit-only`: Display indicator values exclusively at exit signals

These filters remove the `(entry)` and `(exit)` suffixes from indicator names.

### Filtering the trade output by date

Narrow results to a specific period within your backtested range using:

`--timerange : Timerange format YYYYMMDD-[YYYYMMDD]`

Example limiting January output from a full-year backtest:

`freqtrade backtesting-analysis -c <config.json> --timerange 20220101-20220201`

### Printing out rejected signals

Display signals that didn't trigger trades:

`freqtrade backtesting-analysis -c <config.json> --rejected-signals`

### Writing tables to CSV

Export large tables to files instead of terminal output:

`freqtrade backtesting-analysis -c <config.json> --analysis-to-csv`

Output files are written to `user_data/backtest_results` with names like `rejected_signals.csv`, `group_0.csv`, and `group_1.csv`.

Specify alternative output location:

`freqtrade backtesting-analysis -c <config.json> --analysis-to-csv --analysis-csv-path another/data/path/`
# Advanced Hyperopt

This documentation covers sophisticated Hyperopt features requiring strong Python knowledge.

## Custom Loss Functions

To implement a custom loss function, define a `hyperopt_loss_function` in your hyperopt loss class and invoke hyperopt with `--hyperopt-loss YourClassName`.

The function receives these parameters:

- **results**: DataFrame with trade data including columns like `pair`, `profit_ratio`, `profit_abs`, `open_date`, `close_date`, `trade_duration`, and others
- **trade_count**: Number of completed trades
- **min_date** and **max_date**: Timerange boundaries
- **config**: Configuration object (note: hyperopt space parameters may not update here)
- **processed**: Dictionary of DataFrames containing backtesting data by pair
- **backtest_stats**: Statistics in the same format as backtesting output
- **starting_balance**: Initial capital for backtesting

The function must return a float where lower values indicate better results. Since this executes once per epoch, optimization is critical for performance.

## Overriding Pre-Defined Spaces

Create a nested `HyperOpt` class within your strategy to customize optimization spaces. Available overrides include:

- `stoploss_space()`: Custom stop-loss range
- `roi_space()`: Return-on-investment time periods and profit targets
- `generate_roi_table()`: Constructs ROI dictionary from parameters
- `trailing_space()`: Trailing stop configuration
- `max_open_trades_space()`: Simultaneous position limits

All overrides are optional and can be combined as needed.

## Dynamic Parameters

Parameters can be created at runtime within the `bot_start()` callback.

Note: Dynamically created parameters won't appear in strategy parameter counts.

## Custom Base Estimator

Implement `generate_estimator()` in the HyperOpt subclass to specify an Optuna sampler. Supported options include "NSGAIISampler", "TPESampler", "GPSampler", "CmaEsSampler", "NSGAIIISampler", and "QMCSampler". You may also pass an instance inheriting from `optuna.samplers.BaseSampler`.

NSGAIISampler is recommended as the most versatile default choice.

## Space Options

Available space types from `freqtrade.optimize.space`:

- **Categorical**: Choose from discrete options (e.g., `['a', 'b', 'c']`)
- **Integer**: Whole number ranges
- **SKDecimal**: Decimal ranges with defined precision (Freqtrade-specific, preferred option)
- **Real**: Full-precision decimals (rarely necessary, slower)

SKDecimal is recommended over Real for faster optimization with sufficient accuracy.
# Orderflow Data Documentation

## Overview

Freqtrade's orderflow feature enables advanced analysis of public trade data. This capability is currently in beta and may change in future versions.

## Getting Started

### Enable Public Trades

Configure your `config.json` file to activate public trades via `use_public_trades: true` under the `exchange` section.

### Configure Orderflow Processing

Define orderflow settings in your configuration file with these parameters:

- **cache_size**: Number of previous orderflow candles retained in cache
- **max_candles**: Limit for trade data retrieval across candles
- **scale**: Controls price bin sizing for the footprint chart
- **stacked_imbalance_range**: Minimum consecutive imbalanced price levels required
- **imbalance_volume**: Volume threshold for filtering imbalances
- **imbalance_ratio**: Ratio threshold between ask and bid volume

## Downloading Trade Data

Retrieve historical trade data using the download command with the `--dl-trades` flag for backtesting purposes.

## Available Dataframe Columns

Once activated, these columns become accessible:

- **trades**: Individual trade information
- **orderflow**: Footprint chart dictionary
- **imbalances**: Order flow imbalance details
- **bid/ask**: Total volume metrics
- **delta**: Ask-bid volume difference
- **min_delta/max_delta**: Candle extremes
- **total_trades**: Trade count
- **stacked_imbalances_bid/ask**: Price level markers

## Footprint Chart Structure

The orderflow column contains price-indexed data with metrics for each price bin:

- **bid_amount/ask_amount**: Volume at each price level
- **bid/ask**: Order counts
- **delta**: Volume difference
- **total_volume**: Combined volume
- **total_trades**: Trade count

## Raw Trades Data

Individual trade entries include timestamp, date, price, amount, side (buy/sell), unique identifier, and cost.

## Imbalances

Identifies significant ask-bid disparities at specific price levels, marked as boolean bid or ask imbalance values.

## Performance Considerations

Orderflow requires raw trades data which can cause a slow initial startup. Additionally, enabling this feature will cause increased memory usage.
# Advanced Post-installation Tasks

## Overview

This documentation covers advanced configuration and operational tasks for Freqtrade that may be useful in specific environments. These features are optional and primarily intended for users with specific deployment requirements.

## Running Multiple Instances

### Considerations

When operating multiple Freqtrade bots simultaneously on a single machine, three key factors require attention:

- Separate database files for each instance
- Distinct Telegram bot configurations (when Telegram integration is enabled)
- Different port assignments (when REST API is active)

### Database File Management

Freqtrade uses SQLite databases to maintain trade history and position tracking. The system automatically differentiates between live trading and dry-run modes by using separate default database files: `tradesv3.sqlite` for live trading and `tradesv3.dryrun.sqlite` for dry-run operations.

The `--db-url` parameter allows specification of custom database paths using SQLAlchemy URLs. For instance, testing a strategy across multiple stake currencies requires unique database instances for each configuration.

### Docker-based Multi-instance Setup

Running multiple instances via Docker requires editing the docker-compose.yml file to define separate services. Each service should specify unique database URLs, port mappings, and configuration files.

## Alternative Database Systems

Freqtrade supports multiple database backends through SQLAlchemy, including SQLite (default), PostgreSQL, and MariaDB. Users must handle database installation, configuration, and maintenance independently.

### PostgreSQL Configuration

Installation requires psycopg. Connection format: `postgresql+psycopg://<username>:<password>@localhost:5432/<database>`

### MariaDB / MySQL Configuration

Installation requires pymysql. Connection format: `mysql+pymysql://<username>:<password>@localhost:3306/<database>`

## Systemd Service Configuration

Freqtrade can operate as a systemd service by copying the configuration file to the user systemd directory and updating path settings. The system supports watchdog functionality through sd_notify protocol, enabling automatic restart on failure when configured appropriately.

## Advanced Logging Options

Freqtrade provides extensive logging customization through Python's standard logging module. Default behavior includes colored terminal output unless overridden by custom `log_config` settings.

### Syslog Integration

Log messages can be directed to syslog services locally or remotely using the SysLogHandler class. Messages use the user facility and can be filtered or redirected to dedicated log files on systems running rsyslog.

### Journald Integration

The journald system service accepts Freqtrade logs when the `cysystemd` package is installed (unavailable on Windows). Messages can be viewed using `journalctl` with various filtering options.

### JSON Format Logging

Output streams support JSON formatting through custom formatters, allowing structured log data collection and analysis. This format works with both standard output and file handlers.
# Backtesting

Backtesting enables strategy validation using historical data. This feature allows traders to evaluate entry and exit strategies against real market conditions before deploying them.

## Overview

Backtesting requires historical OHLCV (candlestick) data available in `user_data/data/<exchange>`. If data is unavailable for your chosen exchange, pair, and timeframe combination, the system will prompt you to download it first using the data downloading functionality.

The backtesting process confirms whether a strategy has better odds of generating profit versus losses. All calculations include fees using the exchange's default rates.

## Getting Started with Backtesting

### Starting Balance

Backtesting requires an initial capital amount, specified via `--dry-run-wallet <balance>` or `--starting-balance <balance>` command-line arguments, or through the `dry_run_wallet` configuration setting. This amount must exceed `stake_amount` to simulate trades.

### Dynamic Stake Amount

When `stake_amount` is configured as "unlimited," the starting balance splits evenly across `max_open_trades` slots. Profits from earlier trades compound into larger subsequent stake amounts.

## Understanding Results

### Report Tables

**Backtesting Report Table**: Shows all trades made, including position-holding trades at period end. The final totals row displays overall strategy performance, including trade count, average profit percentage, absolute profit, and win rate.

**Left Open Trades Table**: Displays trades forced to close at the backtesting period's end for realistic simulation. These trades appear in both this table and the main report.

**Enter Tag Stats Table**: Breaks down trades by entry tags, showing counts, average profit, total profit, duration, and win/loss distribution for each category.

**Exit Reason Stats Table**: Recaps exit triggers (exit signal, ROI, stoploss, force exit), helping identify which areas require improvement.

**Mixed Tag Stats Table**: Combines entry tags with exit reasons, revealing which entry-exit combinations perform most effectively.

### Summary Metrics

Key performance indicators include:

- **Backtesting Period**: Date range analyzed
- **Trading Mode**: Spot or futures trading designation
- **Trade Statistics**: Total trades, daily averages, counts of wins/draws/losses
- **Capital Metrics**: Starting balance, final balance, absolute and percentage profit
- **Risk Metrics**: Sharpe ratio, Sortino ratio, Calmar ratio, maximum drawdown percentage and duration
- **Quality Indicators**: Profit factor, expectancy ratio, System Quality Number (SQN)
- **Duration Analysis**: Minimum, maximum, and average holding periods for winning and losing positions
- **Streak Data**: Maximum consecutive wins and losses
- **Signal Efficiency**: Rejected entry signals, entry/exit timeouts
- **Best/Worst Performance**: Top and bottom performing pairs and individual trades

### Daily/Weekly/Monthly/Yearly Breakdown

Use `--breakdown <period>` to view time-segmented results. The output displays absolute profit for each period along with trade counts, profit factor, and win/loss distribution that materialized during that timeframe.

## Advanced Features

### Backtest Result Caching

By default, matching strategy and configuration backtests reuse cached results from the previous day. Force a fresh backtest with `--cache none`. Caching disables automatically for open-ended timeranges since underlying data reliability cannot be guaranteed.

### Further Analysis

Backtesting exports trades to file by default, enabling deeper analysis through data frames or web-based visualization. The web interface also supports loading previously generated backtest results for examination without re-running simulations.

### Output Files

Backtesting produces a zip archive containing the backtest report (JSON), market change data, strategy file copy, parameter file (if used), and sanitized configuration file. This ensures result reproducibility with consistent data availability.

### Timeframe Detail Analysis

Append `--timeframe-detail 5m` to simulate intra-candle price movements at finer granularity. This loads both primary and detail timeframe data, evaluating trades at the detail level for greater accuracy on larger timeframes. Detail timeframe must be smaller than the primary timeframe.

## Backtesting Assumptions

Backtesting makes several assumptions about price action within candles:

- Exchange trading limits are respected
- Entry orders fill at the candle's open price unless custom pricing logic applies
- All orders fill at requested prices with no slippage, provided prices stay within the candle's high-low range
- Exit signals trigger at the next candle's open
- Exited positions immediately free trade slots for new pairs
- Exit signal evaluation takes priority over stoploss because signals trigger at candle open
- ROI exits reference the high price but use the ROI value as the actual exit price
- Stoploss exits occur exactly at stoploss price with fees factored in
- Stoploss evaluation precedes ROI within a single candle
- Low price occurs before high for stoploss protection
- Trailing stoploss only adjusts if below the candle's low
- Position reversals (futures) occur when opposing signals trigger on trade-closing candles

### Trading Limits

Exchanges enforce minimum and maximum base and stake currency amounts. Backtesting honors these limits and adjusts stoploss placement accordingly. Historic trading limits are unavailable, potentially inflating requirements when historical prices were higher.

### Precision Limits

Exchange-defined precision limits round or truncate price and amount values. These are based on current limits since historic precision data is unavailable.

## Multiple Strategy Comparison

Test multiple strategies within a single timeframe using `--strategy-list`. All strategies must reside in the same directory unless `--recursive-strategy-search` is enabled. Results include individual strategy details plus a comparative summary table.

## Custom Parameters

Supply custom fee values via `--fee` to account for fee rebates or reductions not visible to the API. This ratio applies twice (entry and exit).

## Timerange Specification

Restrict backtesting data using `--timerange`:
- Until a date: `--timerange=-20180131`
- Since a date: `--timerange=20180131-`
- Between dates: `--timerange=20180131-20180301`
- POSIX timestamps: `--timerange=1527595200-1527618600`

## Next Steps

After confirming profitability, explore hyperparameter optimization to identify ideal parameter values for enhanced performance.
# Freqtrade Basics

This page provides foundational concepts about how Freqtrade operates.

## Freqtrade Terminology

- **Strategy**: Your trading strategy that instructs the bot on actions to take
- **Trade**: An open position
- **Open Order**: An order placed on the exchange that hasn't yet completed
- **Pair**: A tradable currency pair, formatted as Base/Quote (e.g., `XRP/USDT` for spot trading, `XRP/USDT:USDT` for futures)
- **Timeframe**: The candle length used (e.g., `"5m"`, `"1h"`)
- **Indicators**: Technical analysis tools (SMA, EMA, RSI, etc.)
- **Limit Order**: An order that executes at a specified price or better
- **Market Order**: An order guaranteed to fill, though it may impact price based on order size
- **Current Profit**: The unrealized profit on an open trade
- **Realized Profit**: Already captured profit, relevant when using partial exits
- **Total Profit**: Combined realized and unrealized profit, calculated as a percentage against total trade investment

## Fee Handling

"All profit calculations of Freqtrade include fees." During backtesting, dry-run, and hyperopt modes, the exchange's default fee tier is applied. For live trading, actual fees charged by the exchange are used, including any rebates.

## Pair Naming

Freqtrade follows CCXT naming conventions. Using incorrect naming conventions results in the bot failing to recognize pairs and producing errors.

### Spot Pair Naming

Spot pairs use the format `base/quote` (e.g., `ETH/USDT`).

### Futures Pair Naming

Futures pairs use the format `base/quote:settle` (e.g., `ETH/USDT:USDT`).

## Bot Execution Logic

When starting freqtrade in dry-run or live mode, the bot enters a continuous iteration loop that runs every few seconds. Each iteration performs:

1. Fetches open trades from storage
2. Determines the current list of tradable pairs
3. Downloads OHLCV data for pairs and informative pairs (once per candle)
4. Executes the `bot_loop_start()` callback
5. Analyzes the strategy per pair by calling:
   - `populate_indicators()`
   - `populate_entry_trend()`
   - `populate_exit_trend()`
6. Updates trade order states and checks for timeouts
7. Verifies positions and places exit orders considering stoploss, ROI, and exit signals
8. Checks position adjustments for open trades
9. Verifies available trade slots
10. Analyzes entry signals for new positions

## Backtesting / Hyperopt Execution Logic

Backtesting and hyperopt simulate most trading operations with the following process:

1. Load historical data for configured pairs
2. Execute `bot_start()` once
3. Calculate indicators for each pair
4. Calculate entry and exit signals
5. Loop through each candle, simulating entry and exit points:
   - Check order timeouts
   - Evaluate entry signals
   - Confirm trade entries and exits
   - Determine entry prices and leverage
   - Calculate stake amounts
   - Check position adjustments
   - Execute callbacks for filled orders
   - Identify custom exit points
   - Determine exit prices
6. Generate backtest results

Both processes include exchange default fees in calculations.
# Start the Bot

This documentation page explains the parameters and procedures for running the Freqtrade trading bot.

## Overview

The system clock must be accurate and synchronized with NTP servers to ensure proper exchange communication.

## Bot Commands

# Freqtrade Configuration Documentation

## Overview

Freqtrade uses a JSON configuration file to control bot operations. The system supports environment variables, multiple configuration files, and strategy-level overrides with clearly defined precedence rules.

## Configuration File Basics

The bot loads configuration from `config.json` by default, though you can specify a different file using the `-c/--config` command-line option. The configuration format allows standard JSON syntax plus single-line (`//`) and multi-line (`/* */`) comments, along with trailing commas.

Generate a basic configuration with: `freqtrade new-config --config user_data/config.json`

## Environment Variables

Environment variables prefixed with `FREQTRADE__` override configuration file values. Use double underscores (`__`) as section separators following the pattern `FREQTRADE__{section}__{key}`.

Command-line arguments always take precedence over environment variables and configuration files.

## Multiple Configuration Files

You can merge multiple configuration files by specifying them in the `add_config_files` array. Files are resolved relative to the initial configuration file. When keys appear in multiple files, the last specified configuration wins.

This approach helps keep API keys secret by separating them into a private configuration file that overrides values in a shareable main configuration.

## Editor Support

Add this schema line to enable autocomplete and validation in compatible editors:

`"$schema": "https://schema.freqtrade.io/schema.json"`

## Core Configuration Parameters

### Required Settings

**max_open_trades**: Number of simultaneous open trades allowed (or -1 for unlimited, subject to pairlist length).

**stake_currency**: The cryptocurrency used for trading.

**stake_amount**: Amount per trade, or `"unlimited"` to use available balance divided by max open trades. Minimum value is 0.0001.

**dry_run**: Boolean indicating simulation mode (defaults to true).

**minimal_roi**: Dictionary mapping duration in minutes to minimum profit ratios before automatic exit.

**stoploss**: Loss threshold as a ratio (e.g., -0.10 for 10% loss).

**unfilledtimeout.entry**: Minutes/seconds before canceling unfilled entry orders.

**unfilledtimeout.exit**: Minutes/seconds before canceling unfilled exit orders.

**exchange.name**: The exchange class to use.

**strategy**: Strategy class name (recommended via `--strategy` CLI argument).

### Balance and Stake Configuration

**dry_run_wallet**: Starting capital for simulated wallet (defaults to 1000). Accepts float or dictionary for multi-currency simulation.

**tradable_balance_ratio**: Portion of account balance available for trading (defaults to 0.99). Prevents using 100% of funds.

**available_capital**: Starting capital for bots sharing an exchange account, enabling profit compounding per bot without affecting others.

**amend_last_stake_amount**: If true, reduces the final stake to available balance rather than skipping a trade slot.

**last_stake_amount_min_ratio**: Minimum ratio for amended stake amounts (defaults to 0.5).

**amount_reserve_percent**: Reserve percentage for minimum pair stake calculations (defaults to 0.05).

## Pricing Configuration

### Entry Pricing

**entry_pricing.price_side**: Which orderbook side to use: `ask`, `bid`, `same` (for long buys/short sells), or `other` (opposite side).

**entry_pricing.use_order_book**: Enable orderbook-based pricing (defaults to true).

**entry_pricing.order_book_top**: Number indicating orderbook depth to use (defaults to 1 for topmost).

**entry_pricing.price_last_balance**: Interpolation between best side price and last traded price (0.0 = side price, 1.0 = last price).

**entry_pricing.check_depth_of_market.enabled**: Filter entries based on orderbook depth ratios.

**entry_pricing.check_depth_of_market.bids_to_ask_delta**: Threshold for buy-to-sell depth ratio.

### Exit Pricing

**exit_pricing.price_side**: Orderbook side for exits (defaults to `ask` for asks, `bid` for bids).

**exit_pricing.use_order_book**: Enable orderbook-based exit pricing (defaults to true).

**exit_pricing.order_book_top**: Orderbook depth for exits (defaults to 1).

**exit_pricing.price_last_balance**: Interpolation control for exit pricing.

## Order Management

**order_types**: Dictionary mapping actions to order types (market, limit, etc.). Must include `entry`, `exit`, `stoploss`, `emergency_exit`, `force_exit`, `force_entry`, and exchange-specific stoploss settings.

**order_time_in_force**: Specifies order execution policy (`GTC` = Good Till Canceled, `FOK` = Fill Or Kill, `IOC` = Immediate Or Canceled, `PO` = Post Only).

**unfilledtimeout.unit**: Time unit for unfilled timeouts (defaults to `minutes`).

**unfilledtimeout.exit_timeout_count**: Number of exit order timeouts before emergency exit (0 = unlimited).

## Signal and Exit Configuration

**use_exit_signal**: Apply strategy exit signals alongside ROI calculations (defaults to true).

**exit_profit_only**: Only exit when profit exceeds the configured threshold (defaults to false).

**exit_profit_offset**: Minimum profit ratio for exits (defaults to 0.0).

**ignore_roi_if_entry_signal**: Prevent ROI exits while entry signals remain active (defaults to false).

**ignore_buying_expired_candle_after**: Seconds after which buy signals expire within a candle.

**force_entry_enable**: Enable `/forcelong` and `/forceshort` commands (disabled by default for security).

**position_adjustment_enable**: Allow additional buy/sell orders on open positions (defaults to false).

**max_entry_position_adjustment**: Maximum additional orders per trade (-1 for unlimited).

## Exchange Configuration

**exchange.key**: API key (keep secret; required for production).

**exchange.secret**: API secret (keep secret; required for production).

**exchange.password**: API password if required by exchange (keep secret).

**exchange.uid**: API UID if required (keep secret).

**exchange.pair_whitelist**: Trading pairs to monitor and trade.

**exchange.pair_blacklist**: Pairs to exclude from trading.

**exchange.ccxt_config**: Additional CCXT parameters for both sync and async instances.

**exchange.enable_ws**: Enable websocket connections (defaults to true).

**exchange.markets_refresh_interval**: Market data refresh interval in minutes (defaults to 60).

**exchange.skip_open_order_update**: Skip open order updates on startup to avoid exchange issues.

**exchange.unknown_fee_rate**: Fallback fee calculation for non-tradable currency fees.

**exchange.only_from_ccxt**: Prevent data downloads from alternative sources like data.binance.vision.

## Telegram and Notifications

**telegram.enabled**: Enable Telegram bot.

**telegram.token**: Telegram bot token (keep secret).

**telegram.chat_id**: Your Telegram account ID (keep secret).

**telegram.balance_dust_level**: Minimum balance to show in `/balance` command.

**telegram.reload**: Allow reload buttons in messages (defaults to true).

**telegram.allow_custom_messages**: Allow strategies to send custom Telegram messages.

## Webhook Configuration

**webhook.enabled**: Enable webhook notifications.

**webhook.url**: Webhook endpoint URL.

**webhook.entry/entry_cancel/entry_fill/exit/exit_cancel/exit_fill/status**: Payload templates for different trade events.

**webhook.allow_custom_messages**: Allow strategies to send custom webhook messages.

## API Server

**api_server.enabled**: Enable REST API and FreqUI support.

**api_server.listen_ip_address**: Bind IP address for API server.

**api_server.listen_port**: Bind port (1024-65535).

**api_server.verbosity**: Log level (`info` or `error`).

**api_server.username**: API server username (keep secret).

**api_server.password**: API server password (keep secret).

**api_server.ws_token**: WebSocket token (keep secret).

## Additional Parameters

**fiat_display_currency**: Fiat currency for profit reporting (supports major fiat currencies and crypto like BTC, ETH, XRP, LTC, BCH, BNB).

**fee**: Trading fee for backtesting (normally left as default for exchange fee).

**futures_funding_rate**: User-specified funding rate for leverage trading when historical rates unavailable.

**trading_mode**: `spot` (default), `margin`, or `futures`.

**margin_mode**: `cross` or `isolated` for leverage trading.

**liquidation_buffer**: Safety margin between stoploss and liquidation price (defaults to 0.05).

**timeframe**: Candle interval (e.g., 1m, 5m, 15m, 1h) - usually in strategy.

**process_only_new_candles**: Only process indicators when new candles arrive (defaults to true).

**custom_price_max_distance_ratio**: Maximum distance for custom entry/exit prices (defaults to 0.02).

**cancel_open_orders_on_exit**: Cancel unfilled orders on `/stop` or bot termination (defaults to false).

**initial_state**: Bot startup state (`running`, `paused`, or `stopped`).

**disable_dataframe_checks**: Skip dataframe validation (defaults to false).

**internals.process_throttle_secs**: Minimum loop duration in seconds (defaults to 5).

**internals.heartbeat_interval**: Heartbeat message interval in seconds (defaults to 60; 0 to disable).

**internals.sd_notify**: Enable systemd service notifications.

**db_url**: Database connection string (auto-configured based on dry_run setting).

**logfile**: Log file path with rolling rotation.

**dataformat_ohlcv**: Format for candle storage (defaults to `feather`).

**dataformat_trades**: Format for trade data storage (defaults to `feather`).

**reduce_df_footprint**: Convert numeric columns to float32/int32 to save memory (defaults to false).

**strategy_path**: Additional directory for strategy files.

**recursive_strategy_search**: Search strategy subdirectories (defaults to false).

**user_data_dir**: User data directory location (defaults to `./user_data/`).

**pairlists**: Pairlist handler configuration.

**coingecko.api_key**: Coingecko API key for fiat conversion (optional).

**coingecko.is_demo**: Use demo Coingecko API key.

## Configuration Precedence

1. CLI arguments (highest priority)
2. Environment variables
3. Configuration files (last file wins)
4. Strategy definitions (lowest priority)

## Dry-Run Mode Considerations

The simulated wallet in dry-run mode assumes trades based on orderbook depth for market orders (with maximum 5% slippage) and price-based fills for limit orders. Stoploss prices are assumed filled when using `stoploss_on_exchange`. Open orders persist across restarts with the assumption they weren't filled while offline.

## Production Mode Setup

Switch from dry-run by setting `dry_run: false` and inserting exchange API credentials. Use separate configuration files to keep credentials secret and avoid sharing sensitive keys. Consider using a fresh database separate from dry-run data to maintain clean trading statistics.

## Proxy Configuration

Standard HTTP/HTTPS proxies use environment variables `HTTP_PROXY` and `HTTPS_PROXY`. Exchange requests require CCXT-specific proxy configuration through `exchange.ccxt_config` with properties like `httpsProxy` and `wsProxy`.

## WebSocket Support

Freqtrade supports CCXT.pro websockets with fallback to REST API if connections fail. Disable via `exchange.enable_ws: false` if problematic. Rollout is gradual and currently limited to OHLCV data streams on select exchanges.

# Data Downloading

## Getting data for backtesting and hyperopt

To download candles (OHLCV) needed for backtesting and hyperoptimization, use the `freqtrade download-data` command.

By default, freqtrade downloads data for 1-minute and 5-minute timeframes covering the last 30 days. Exchange and pair information comes from your configuration file (if specified with `-c/--config`). Without a configuration, `--exchange` becomes required.

You can specify relative timeranges using `--days 20` or absolute starting points with `--timerange 20200101-`. For incremental downloads, the relative approach is recommended.

### Updating existing data

If you already have backtesting data and want to refresh it, freqtrade automatically calculates the missing timerange. The download begins from the latest available point through today. Neither `--days` nor `--timerange` parameters are required. When adding new pairs to existing data, use `--new-pairs-days xx` to download that many days for new pairs while updating old pairs with only missing data.

### Usage

The command supports numerous options including:

- `--pairs`: Specify currency pairs (space-separated)
- `--pairs-file`: Use pairs from a file
- `--days`: Number of days to download (default: 30)
- `--new-pairs-days`: Days to download for new pairs
- `--timerange`: Absolute timerange specification
- `--dl-trades`: Download trades instead of OHLCV data
- `--timeframes`: Specify which timeframes to download
- `--data-format-ohlcv`: Storage format (json, jsongz, feather, parquet)
- `--data-format-trades`: Trade data format
- `--trading-mode`: Select spot, margin, or futures
- `--candle-types`: Specify candle types for futures
- `--prepend`: Allow data prepending

### Other Notes

- Use `--datadir user_data/data/some_directory` for a custom directory
- Change exchanges with `--exchange <exchange>` or use a different configuration file
- Use `--pairs-file some_other_dir/pairs.json` for alternative pairs files
- Download data for specific durations with `--days 10`
- Use `--timerange 20200101-` for downloads from fixed starting points
- Specify timeframes with `--timeframes 1m 5m`
- Leverage configuration files with `-c/--config`
- When downloading futures data, necessary candle types are automatically included

### Download additional data before the current timerange

If you previously downloaded 2022 data but now need earlier data, use the `--prepend` flag with an end-date. Freqtrade ignores the end-date if data exists, updating it to match the existing data's start point.

### Data format

Freqtrade supports four data formats:

- **feather**: Apache Arrow-based format
- **json**: Plain text JSON files
- **jsongz**: Gzip-compressed JSON
- **parquet**: Columnar datastore (OHLCV only)

Default format for both OHLCV and trades data is feather. Specify formats with `--data-format-ohlcv` and `--data-format-trades` command-line arguments. You can convert between formats using convert-data and convert-trade-data commands.

#### Dataformat comparison

Performance comparisons show:

| Format | Size | Timing |
|--------|------|--------|
| feather | 72Mb | 3.5s |
| json | 149Mb | 25.6s |
| jsongz | 39Mb | 27s |
| parquet | 83Mb | 3.8s |

For optimal performance and file size, feather or parquet formats are recommended.

### Pairs file

Create a `pairs.json` file in your data directory listing each pair. The pairs file is used only when no configuration is loaded. You can force its usage with `--pairs-file pairs.json`, though using configuration-based pairlists is recommended.

## Sub-command convert data

Convert between OHLCV data formats. The conversion can erase originals with `--erase`.

## Sub-command convert trade data

Convert trade data between formats.

## Sub-command trades to ohlcv

When using `--dl-trades` (primarily for Kraken), convert trade data to OHLCV for additional timeframes without re-downloading.

## Sub-command list-data

View downloaded data. This displays available pair and timeframe combinations. Use `--show-timerange` to see available date ranges for each dataset.

## Trades (tick) data

Most exchanges provide historic trade data via API, useful when you need multiple timeframes. Data is downloaded once and resampled locally. Files use feather format by default and follow the naming convention `<pair>-trades.feather`.

Incremental mode is supported for trades data. Add `--dl-trades` to your command to use this mode. With `--convert`, resampling happens automatically and overwrites existing OHLCV data.

Note: This approach should primarily be used by Kraken users, as most exchanges provide sufficient OHLCV history. Kraken Futures uses standard OHLCV downloads without requiring `--dl-trades`. Async calls make this method slower since each request depends on the previous result.

# Deprecated Features - Freqtrade

This page documents command line arguments, configuration parameters, and bot features that are no longer supported.

## Removed Features

### the `--refresh-pairs-cached` command line option

This option, used in backtesting and hyperopt contexts to refresh candle data, was deprecated in 2019.7-dev and removed in 2019.9. The functionality has been replaced by the separate `freqtrade download-data` sub-command.

### The `--dynamic-whitelist` command line option

Deprecated in 2018 and removed in freqtrade 2019.6-dev and 2019.7. Users should refer to pairlists instead.

### the `--live` command line option

This backtesting option downloaded only the latest 500 candles, making it ineffective for quality backtest data. It was removed in 2019.7-dev and freqtrade 2019.8.

### `ticker_interval` (now `timeframe`)

The terminology `ticker_interval` was deprecated in 2020.6 in favor of `timeframe`. Compatibility code was removed in 2022.3.

### Allow running multiple pairlists in sequence

The configuration section `"pairlist"` has been replaced by `"pairlists"` (a list for specifying sequential pairlists). The old section was deprecated in 2019.11 and removed in 2020.4.

### deprecation of bidVolume and askVolume from volume-pairlist

Since only quoteVolume is comparable between assets, `bidVolume` and `askVolume` were deprecated in 2020.4 and removed in 2020.9.

### Using order book steps for exit price

The `order_book_min` and `order_book_max` parameters for stepping the orderbook were removed in 2021.7 due to increased risk and maintenance concerns.

### Legacy Hyperopt mode

Separate hyperopt files were deprecated in 2021.4 and removed in 2021.9. Users should switch to Parametrized Strategies.

## Strategy Changes Between V2 and V3

Isolated Futures and short trading introduction in 2022.4 required major changes. For existing spot market strategies, no immediate changes are necessary, though the development team will announce any future deprecation separately with an appropriate transition period.

### webhooks - changes with 2022.4

#### `buy_tag` has been renamed to `enter_tag`

A compatibility layer will be maintained for 1-2 versions, supporting both terms. However, webhook support for the old terminology will eventually disappear.

#### Naming Changes

Webhook terminology shifted from "sell" to "exit" and "buy" to "entry":

- `webhookbuy`, `webhookentry` → `entry`
- `webhookbuyfill`, `webhookentryfill` → `entry_fill`
- `webhookbuycancel`, `webhookentrycancel` → `entry_cancel`
- `webhooksell`, `webhookexit` → `exit`
- `webhooksellfill`, `webhookexitfill` → `exit_fill`
- `webhooksellcancel`, `webhookexitcancel` → `exit_cancel`

## Removal of `populate_any_indicators`

Version 2023.3 removed `populate_any_indicators` in favor of split methods for feature engineering and targets. See the strategy migration document for details.

## Removal of `protections` from configuration

Configuration-based protections via `"protections": []` were removed in 2024.10 after deprecation warnings lasting over 3 years.

## hdf5 Data Storage

HDF5 data storage was deprecated in 2024.12 and removed in 2025.1. Users should switch to feather format and use the `convert-data` subcommand to migrate existing data.

## Configuring Advanced Logging via Config

Configuring syslog and journald via `--logfile systemd` and `--logfile journald` was deprecated in 2025.3. Use configuration-based log setup instead.

## Removal of the Edge Module

The edge module was deprecated in 2023.9 and removed in 2025.6. Having edge configured will result in an error.

## Adjustment to Dynamic Funding Rate Handling

Version 2025.12 adjusted dynamic funding rate handling to support 1h funding intervals. Mark and funding rate timeframes changed from 8h to 1h for all supported futures exchanges.

### Strategy

Most strategies require no adjustments, but those using `@informative("8h", candle_type="funding_rate")` must switch to 1h. The same applies to `dp.get_pair_dataframe()` calls with similar parameters. Freqtrade will auto-adjust with a warning.

### Selective Data Re-download

Users can clean up old data and selectively download new mark and funding rate data for their specific exchange and timeframe needs rather than re-downloading everything.

**Hyperliquid Exception**: No 1h mark data is required—regular candles are used instead.

## Catboost Models in FreqAI

CatBoost models were removed in version 2025.12. Existing implementations can be copied from git history, but users should migrate to LightGBM or XGBoost for better support.

## Error Handling

## Plugins

### Pairlists

Develop new pair selection algorithms by examining the VolumePairList handler and creating a similar implementation. The base class provides exchange instances, pairlist manager access, and configuration objects. Register new pairlists in `constants.py` under `AVAILABLE_PAIRLISTS`.

Configuration occurs in the bot's configuration file under the `pairlists` element. By convention, use `number_assets` to specify maximum pair count, ensuring consistent user experience.

The `short_desc()` method returns descriptions for Telegram messages in the format "PairlistName - top/bottom X pairs". The `gen_pairlist()` method enables leading pairlist functionality, defining initial pairlists processed by subsequent handlers. This method is called each iteration; implement caching for expensive calculations.

The `filter_pairlist()` method processes pairlists through the handler chain. It receives a pairlist and pre-fetched ticker data. The default implementation validates each pair, but can be overridden. It must return the resulting pairlist.

### Protections

Read the protection documentation to understand these systems. New protections must inherit from `IProtection` and implement `short_desc()`, `global_stop()`, and `stop_per_pair()` methods. These methods return ProtectionReturn objects containing lock pair status, lock duration, reason string, and lock side designation.

Protections should use `calculate_lock_end()` for duration calculation and employ `stop_duration`/`stop_duration_candles` configuration parameters. Per-pair protections set `has_local_stop=True` and trigger on trade closure. Global protections set `has_global_stop=True` and evaluate across all pairs, locking all from trading.

Lock end times should be calculated based on the final trade considered, preventing re-locking if the lookback period exceeds the lock duration.

## Implementing New Exchanges

Most CCXT-supported exchanges work out of the box. Custom implementations belong in the `freqtrade/exchange` source folder and require importing in `freqtrade/exchange/__init__.py`. Implementing exchanges involves trial and error and requires development experience.

Test public endpoints by configuring exchanges in `tests/exchange_online/conftest.py` and running `pytest --longrun tests/exchange_online/test_ccxt_compat.py`. Also use `freqtrade download-data` for extended timeranges to verify correct data retrieval.

Additional testing includes verifying `fetch_ohlcv()` data, checking L2 orderbook limits, confirming balance display, creating and canceling orders, and completing full trades while comparing exchange and bot calculations.

### Stoploss On Exchange

Verify whether exchanges support Stoploss on Exchange orders through their API. Since CCXT lacks unified support, implement exchange-specific parameters directly. The binance.py file provides an example.

### Incomplete Candles

Exchanges may return incomplete candles. Use the provided snippet to test exchange behavior by fetching OHLCV data and comparing the last entry date with the current UTC date. If they match, the candle is incomplete and should be dropped by leaving `ohlcv_partial_candle` unchanged. Otherwise, set it to `False`.

### Updating Binance Leverage Tiers

Leverage tiers require periodic updates with an authenticated account having futures enabled. The process involves loading markets, fetching leverage tiers via the CCXT exchange object, and saving results to `freqtrade/exchange/binance_leverage_tiers.json`.

## Troubleshooting

### Docker on Windows

**Error: "Timestamp for this request is outside of the recvWindow."**

The market api requests require a synchronized clock but the time in the docker container shifts a bit over time into the past. To fix this issue temporarily you need to restart docker. A permanent solution is either to host the docker container on a linux host or restart the wsl from time to time with the scheduler.

**Cannot connect to the API (Windows)**

If you're on windows and just installed Docker (desktop), make sure to reboot your System. Docker can have problems with network connectivity without a restart.

**Warning**

Due to the above, we do not recommend the usage of docker on windows for production setups, but only for experimentation, datadownload and backtesting. Best use a linux-VPS for running freqtrade reliably.
# Exchange-specific Notes

This documentation covers exchange-specific configuration and gotchas for Freqtrade, a cryptocurrency trading bot.

## Quick Overview of Supported Exchange Features

The page provides a table comparing multiple exchanges (Binance, Bingx, Bitmart, Bitget, Bybit, Gate.io, HTX, Hyperliquid, Kraken, OKX, Bitvavo, Kucoin) across trading modes (spot/futures), margin modes, and supported stoploss types.

## Exchange Configuration

Freqtrade relies on the CCXT library, supporting over 100 cryptocurrency exchanges. Basic configuration requires exchange name, API key, and secret.

Rate limits can be adjusted through `ccxt_config` and `ccxt_async_config` settings. The documentation notes that "optimal settings for rate-limiting depend on the exchange and the size of the whitelist, so an ideal parameter will vary."

## Binance

Binance restricts API access by geographic location, blocking certain countries including Canada, Malaysia, Netherlands, and the United States.

The exchange supports stoploss on exchange functionality using stop-loss-limit orders for spot trading and both stop-limit and stop-market for futures.

Binance users should add "BNB/<STAKE>" to their blacklist unless maintaining extra BNB for fees, as "Binance accounts may use BNB for fees, and if a trade happens to be on BNB, further trades may consume this position."

Two separate Binance platforms exist: binance.com for international users and binance.us for US-based traders.

Futures trading requires orderbook-based entry and exit pricing. Isolated futures mode requires "One-way Mode" position settings and "Single-Asset Mode" configuration.

## Other Major Exchanges

**Bingx**: Supports stoploss on exchange with both stop-limit and stop-market orders.

**Kraken**: Requires `--dl-trades` for backtesting due to providing only 720 historic candles. The exchange supports stoploss on exchange and manual rate limit configuration.

**Kucoin**: Requires an API passphrase in configuration. Supports stoploss on exchange and recommends blacklisting "KCS/<STAKE>" to prevent fee-related trading issues.

**OKX**: Requires API passphrase. Only provides 100 candles per API call, limiting backtesting data. Futures mode supports "Buy/Sell" or hedge positioning.

**Gate.io**: Supports stoploss on exchange with stop-loss-limit orders.

**Bybit**: Supports stoploss on exchange for futures (isolated mode). Requires position mode set to "One-way Mode" on startup.

**Bitget**: Requires API passphrase. Supports stoploss on exchange with both stop-loss-market and stop-loss-limit orders.

**Hyperliquid**: A decentralized exchange requiring wallet address and private key authentication. Does not support market orders and provides only 5000 historic candles. Supports vaults and subaccounts for isolated trading environments.

## General Considerations

Invalid Nonce errors typically require API key regeneration rather than nonce reset.

Incomplete candles are typically fetched by exchanges but removed by Freqtrade to prevent repainting issues. The documentation recommends using data provider functionality within strategies for latest price information.

Advanced configuration options can override defaults through `_ft_has_params` settings, though the documentation warns these customizations may cause unexpected behavior.
# Freqtrade FAQ

### Can freqtrade open multiple positions on the same pair in parallel?

No, only one position per pair at a time. Use the `adjust_trade_position()` callback to modify open positions.

### Does freqtrade support sandbox accounts?

No, but dry-run mode simulates trading without real funds. Sandbox markets lack realistic order books and liquidity.

### The bot does not start

If `freqtrade: command not found` appears, activate the virtual environment with `source .venv/bin/activate` or verify installation completion.

### The bot starts, but in STOPPED mode

Set `"initial_state": "running"` in your config.json.

### I have waited 5 minutes, why hasn't the bot made any trades yet?

Trade entry depends on strategy signals, coin selection, and market conditions. Backtesting shows expected trade frequency but not distribution. Check logs for errors or missing buy signals.

### I have made 12 trades already, why is my total profit negative?

Twelve trades provide insufficient data for assessment. Backtesting demonstrates profitability over thousands of trades, though individual coins may show losses.

### I'd like to make changes to the config. Can I do that without having to kill the bot?

Yes, edit your config and use `/reload_config` to reload configuration and strategy without stopping the bot.

### Why does my bot not sell everything it bought?

This "coin dust" occurs when exchanges subtract fees from received currency amounts, leaving fractional amounts below minimum tradable lot sizes. Using exchange fee currencies (like BNB on Binance) can mitigate this issue.

### I deposited more funds to the exchange, but my bot doesn't recognize this

Freqtrade updates balances before placing orders and via RPC calls once hourly. Use `/reload_config` to force immediate updates.

### I want to use incomplete candles

Freqtrade doesn't provide incomplete candles to strategies, as this causes repainting and unrealistic backtests. Use dataprovider orderbook or ticker methods for current market data, though these don't work during backtesting.

### Is there a setting to only Exit the trades being held and not perform any new Entries?

Use `/stopentry` in Telegram to prevent new entries, followed by `/forceexit all` to sell open positions.

### I sold the bot's capital and now there's errors in the log

Freqtrade assumes sole management of its trades. Selling bot capital triggers recovery attempts through on-exchange order searches, though this best-effort approach may fail with unsupported order types or older trades.

### I want to run multiple bots on the same machine

Consult the advanced setup documentation for multiple instance guidance.

### I'm getting "Impossible to load Strategy" when starting the bot

Verify the strategy name matches the class name (case-sensitive), the file is in `user_data/strategies` with `.py` extension, dependencies are installed, and (in Docker) the strategy directory is mounted correctly.

### I'm getting "Missing data fillup" messages in the log

This warning indicates missing candles, common on low-volume pairs without trades in the specified timeframe. Freqtrade fills these with empty candles matching previous closes.

### I'm getting "Price jump between 2 candles detected"

This warning signals >30% price jumps, possibly from token exchanges or trading halts. Often accompanied by missing data fillup messages.

### I want to reset the bot's database

Delete the default database (`tradesv3.sqlite` or `tradesv3.dryrun.sqlite`) or specify a new URL with `--db-url`.

### I'm getting "Outdated history for pair xxx" in the log

The bot received outdated candles and won't enter trades. Causes include exchange downtime, incorrect system time, low-volume pairs, or API issues. Verify system time and pair trading volume.

### I get the message "Couldn't reuse watch for xxx" in the log

This informational message indicates websocket candles unavailable, usually from connection interruptions or zero-trade timeframes. Freqtrade falls back to REST API gracefully.

### I'm getting the "Exchange XXX does not support market orders." message

Your exchange requires limit orders instead of market orders. Modify order types in strategy or config.

### I'm trying to start the bot live, but get an API permission error

Invalid or expired API keys, disabled IPs, or missing permissions cause these errors. Verify key validity, remove leading/trailing spaces, enable required exchange permissions, and ensure your IP is whitelisted.

### How do I search the bot logs for something?

Redirect stderr to search logs. Use standard shell redirection on Unix shells or write logs to a file with `--logfile` for persistent searching.

## Hyperopt Module

### Why does freqtrade not have GPU support?

Most indicator libraries lack GPU support, limiting GPU benefits to pandas operations. GPUs excel at calculations but struggle with mixed computational and Python code execution required by hyperopt.

### How many epochs do I need to get a good Hyperopt result?

Default 100 epochs is insufficient. Run 10,000 total epochs (500-1000 at a time) to discover optimal strategies. Monitor results—continued improvements indicate further optimization potential.

### Why does it take a long time to run hyperopt?

Discovering effective strategies requires extensive computation. Search space grows exponentially with triggers, guards, and stoploss combinations. Runtime depends on CPU, storage, RAM, timeframe, pair count, and trade frequency.

A FreqAI config must minimally include these parameters (under a `freqai` section):

- `purge_old_models`: number

## Building a FreqAI strategy

A FreqAI strategy requires specific methods in the standard Freqtrade strategy structure:

- `populate_indicators()` - calls `self.freqai.start()`
- `feature_engineering_expand_all()` - auto-expands features across configured timeframes and periods
- `feature_engineering_expand_basic()` - expands features without multiplying across indicator periods
- `feature_engineering_standard()` - final feature engineering, good for non-auto-expanding features
- `set_freqai_targets()` - defines labels for the model

Features must be prepended with `%` to be recognized. Use `%%` to return features for plotting. Targets must be prepended with `&`.

Note: `self.freqai.start()` can only be called in `populate_indicators()`, and features must be defined in `feature_engineering_*()` methods, not in `populate_indicators()`.

| `df['&*']` | Training targets/labels set in `set_freqai_targets()` |

# FreqAI Developer Guide

## Project Architecture

FreqAI's architecture comprises three distinct objects designed to encourage development of unique features and models:

- **IFreqaiModel**: A persistent object containing logic to collect, store, and process data, engineer features, and run training and inference models.
- **FreqaiDataKitchen**: A non-persistent object created uniquely for each asset/model, containing metadata and various data processing tools.
- **FreqaiDataDrawer**: A persistent object containing historical predictions, models, and save/load methods.

Built-in prediction models inherit directly from `IFreqaiModel` and have access to all parent methods. Advanced users typically override `fit()`, `train()`, `predict()`, and `data_cleaning_train/predict()` functions.

## Data Handling

FreqAI organizes model files, prediction data, and metadata in the `user_data_dir/models/` directory. This structure enables post-processing and crash resilience through automatic data reloading. The `FreqaiDataKitchen()` depends heavily on this file structure and should not be manually modified.

### File Structure

The directory structure is automatically generated based on the model identifier specified in configuration:

| File/Folder | Purpose |
|---|---|
| `config_*.json` | Copy of model-specific configuration |
| `historic_predictions.pkl` | All historic predictions during model deployment; enables reload after crashes |
| `pair_dictionary.json` | Training queue and location of recently trained model |
| `sub-train-*_TIMESTAMP` | Folder containing single model files including metadata, model file, PCA object, SVM model, training dataframe, and associated dates |

Each `sub-train-` folder contains normalized data, feature lists, and optional dimensionality reduction transforms for handling unseen prediction features.
# Feature Engineering

## Defining the Features

Low level feature engineering is performed in the user strategy within a set of functions called `feature_engineering_*`. These functions set the base features such as RSI, MFI, EMA, SMA, time of day, volume, etc. The base features can be custom indicators or they can be imported from any technical-analysis library.

FreqAI is equipped with a set of functions to simplify rapid large-scale feature engineering:

| Function | Description |
|----------|-------------|
| `feature_engineering_expand_all()` | Automatically expands defined features on config-defined `indicator_periods_candles`, `include_timeframes`, `include_shifted_candles`, and `include_corr_pairs`. |
| `feature_engineering_expand_basic()` | Automatically expands features on `include_timeframes`, `include_shifted_candles`, and `include_corr_pairs` but does not expand across `indicator_periods_candles`. |
| `feature_engineering_standard()` | Called once with the base timeframe dataframe after all other feature engineering functions have run. Ideal for custom exotic feature extractions and features that should not be auto-expanded. |
| `set_freqai_targets()` | Required function to set model targets, which must be prepended with `&`. |

It is advisable to start from the template `feature_engineering_*` functions in the example strategy to ensure correct conventions.

High level feature engineering is handled within `"feature_parameters":{}` in the FreqAI config. Within this file, users can decide large scale feature expansions on top of base features such as including correlated pairs, including informative timeframes, or including recent candles.

Features must be prepended with `%` to be recognized by FreqAI internals, while targets must be prepended with `&`.

The `include_timeframes` in the config are the timeframes of each call to `feature_engineering_expand_*()` in the strategy. Users can ask for each defined feature to be included for informative pairs using the `include_corr_pairlist`.

`include_shifted_candles` indicates the number of previous candles to include in the feature set. The total number of features created equals: length of `include_timeframes` × number of features × length of `include_corr_pairlist` × number of `include_shifted_candles` × length of `indicator_periods_candles`.

### Gain Finer Control Over `feature_engineering_*` Functions With Metadata

All `feature_engineering_*` and `set_freqai_targets()` functions are passed a `metadata` dictionary containing information about the pair, timeframe, and period that FreqAI is automating for feature building. Users can use `metadata` as criteria for blocking or reserving features for certain timeframes, periods, or pairs.

### Returning Additional Info From Training

Important metrics can be returned to the strategy at the end of each model training by assigning them to `dk.data['extra_returns_per_train']['my_new_value'] = XYZ` inside the custom prediction model class. FreqAI expands these values to fit the returned dataframe.

Users need to set the standard dictionary in the config so that FreqAI can return proper dataframe shapes. These values will likely be overridden by the prediction model, but serve as default initial values if needed.

### Weighting Features for Temporal Importance

FreqAI allows setting a `weight_factor` to weight recent data more strongly than past data via an exponential function. This emphasizes temporal patterns in the training data.

## Building the Data Pipeline

By default, FreqAI builds a dynamic pipeline based on user configuration settings. The default settings include a `MinMaxScaler(-1,1)` and a `VarianceThreshold` which removes any column with zero variance.

Users can activate additional pipeline steps through configuration parameters:
- `use_SVM_to_remove_outliers: true` adds the SVMOutlierExtractor
- `principal_component_analysis: true` activates PCA
- `DI_threshold: 1` activates the DissimilarityIndex
- `noise_standard_deviation: 0.1` adds noise to data
- `use_DBSCAN_to_remove_outliers: true` adds DBSCAN outlier removal

### Customizing the Pipeline

Users can customize the data pipeline by setting `dk.feature_pipeline` to their desired Pipeline object inside their `IFreqaiModel` `train()` function, or by overriding `define_data_pipeline` and `define_label_pipeline` functions.

FreqAI uses the DataSieve pipeline, which follows the SKlearn pipeline API but adds coherence between X, y, and sample_weight vector point removals, feature removal, and feature name tracking.

Users can wrap most SKLearn transformation steps in the `SKLearnWrapper` class or use transformations from the DataSieve library. Custom transformations can be created by inheriting from the datasieve `BaseTransform` and implementing `fit()`, `transform()`, and `inverse_transform()` methods.

### Migrating a Custom `IFreqaiModel` to the New Pipeline

If a custom `IFreqaiModel` relies on `data_cleaning_train/predict()`, migration to the new pipeline is necessary. If the model does not rely on these functions, no migration is needed.

## Outlier Detection

Equity and crypto markets suffer from high levels of non-patterned noise in the form of outlier data points. FreqAI implements various methods to identify and mitigate such outliers.

### Identifying Outliers With the Dissimilarity Index (DI)

The Dissimilarity Index quantifies the uncertainty associated with each prediction made by the model. Users can remove outlier data points by including `"DI_threshold": 1` in the feature_parameters config.

The DI measures the distance between each training data point and all other training data points, establishing a characteristic distance. This is compared to the distance between a new prediction feature vector and training data to estimate whether the prediction is an outlier.

A higher `DI_threshold` allows predictions further from the training data, while a lower threshold discards more predictions.

### Identifying Outliers Using a Support Vector Machine (SVM)

Users can tell FreqAI to remove outlier data points by including `"use_SVM_to_remove_outliers": true` in feature_parameters. The SVM is trained on training data and removes any data points deemed beyond the feature space.

Users can provide additional SVM parameters such as `shuffle` and `nu` via the `feature_parameters.svm_params` dictionary. The parameter `shuffle` is by default set to `False` for consistent results, while `nu` broadly represents the amount of data points considered outliers (0 to 1).

### Identifying Outliers With DBSCAN

Users can activate `use_DBSCAN_to_remove_outliers` in the config to use DBSCAN for clustering and removing outliers. This unsupervised machine learning algorithm clusters data without needing to know the number of clusters beforehand.

DBSCAN identifies core points that have a specified number of other data points within a distance threshold, edge points within that distance from core points, and outliers that fall outside clusters. FreqAI uses `sklearn.cluster.DBSCAN` with `min_samples` set to one-quarter of the number of candles in the feature set and `eps` computed automatically as the elbow point in the k-distance graph.

### Data Dimensionality Reduction With Principal Component Analysis

Users can reduce feature dimensionality by activating `principal_component_analysis: true` in feature_parameters. PCA reduces dimensionality so that the explained variance is greater than or equal to 0.999. Reducing dimensionality makes model training faster and allows for more up-to-date models.

# Reinforcement Learning in Freqtrade

## Background and Terminology

### What is RL and Why Does FreqAI Need It?

Reinforcement learning operates through two core components: an agent and a training environment. The agent progresses through historical market data, selecting from five possible actions: long entry, long exit, short entry, short exit, or neutral positioning. The environment evaluates performance and provides rewards according to a custom `calculate_reward()` function.

State information forms a second critical component, feeding current profit, position status, and trade duration into the network at each step. This information trains the agent during the training phase and reinforces it in dry-run and live environments (unavailable in backtesting). FreqAI and Freqtrade integrate naturally with this reinforcing mechanism since this data is readily accessible in live deployments.

Reinforcement learning represents an evolution for FreqAI by adding adaptivity and market reactivity beyond what classifiers and regressors offer. However, classical models provide robust predictions that RL agents may lack. Improperly trained RL agents can discover "shortcuts" to maximize reward without generating profitable trades, making RL more complex than typical machine learning approaches.

### The RL Interface

The training environment is exposed through a prediction model file—a user-inherited `BaseReinforcementLearner` object customized via `MyRLEnv`. Most users focus on designing creative `calculate_reward()` functions while leaving the environment unchanged. Other users may only adjust configuration settings and leverage existing feature engineering. Advanced users can create custom model classes entirely.

The framework builds on stable_baselines3 (torch) and OpenAI gym. The model class remains well-isolated, enabling competing libraries to integrate easily. The environment inherits from `gym.Env`, requiring entirely new environment implementations to switch libraries.

### Important Considerations

The RL training environment differs significantly from real backtesting environments. It represents a simplified market without complex strategy logic like callbacks, custom exit conditions, or leverage controls. The agent learns its policy (stoploss, take profit rules) entirely through the `calculate_reward()` function. Consequently, training environments are not identical to real-world conditions.

## Running Reinforcement Learning

Setting up an RL model matches the process for regressors or classifiers using the same command-line flags.

The strategy follows standard feature engineering patterns, but differs in target creation. RL requires no explicit labels. Instead, a neutral action value must be set in the action column.

Raw price data (open, high, low, close) must pass to the agent for training environment access. The `&-action` column contains agent actions for determining entries and exits in the strategy. Action values align with the environment used—FreqAI provides two default environments, both using 0 as the neutral action.

## Configuring the Reinforcement Learner

The `rl_config` dictionary must exist within the `freqai` configuration. Key parameters:

- `train_cycles` determines how many times the agent cycles through candle data for training.
- `model_type` selects from available stable_baselines models.
- Enabling `continual_learning` in the main FreqAI configuration allows the RL library to continue training new models from previous model states rather than retraining from scratch.
- The `model_training_parameters` dictionary should contain hyperparameter customizations specific to the selected `model_type`.

## Creating a Custom Reward Function

The provided reward function demonstrates functionality and runs quickly on modest computers, but is unsuitable for production. Users must create custom `calculate_reward()` functions or use templates from community sources, saved to `user_data/freqaimodels`.

The `calculate_reward()` function, located within the `MyRLEnv` class, expresses creative market theories. Examples include rewarding winning trades while penalizing losing ones, or rewarding entries while penalizing prolonged trade duration.

Optimal reward functions are continuously differentiable and well-scaled. Single large penalties for rare events prevent effective learning. Small penalties applied to common events enable faster agent development. Scaling penalties with severity (using linear or exponential functions) improves reward continuity more effectively than sudden point penalties.

## Using Tensorboard

RL models benefit from tracking training metrics. FreqAI integrates Tensorboard for monitoring training and evaluation performance. Run `tensorboard --logdir user_data/models/unique-id` in a separate terminal shell, then access output at 127.0.0.1:6006 (Tensorboard's default port).

## Custom Logging

FreqAI provides a built-in episodic summary logger called `self.tensorboard_log` for adding custom information to Tensorboard logs. This function records agent actions once per training step. Values accumulate throughout each episode and report upon conclusion, then reset for the next episode.

`self.tensorboard_log` works anywhere within the environment. For example, it can track how frequently various reward components are triggered. The function tracks incremented objects (events, actions). For float metrics, pass the value as a second argument—values remain unchanged rather than incrementing.

## Choosing a Base Environment

FreqAI provides three base environments: `Base3ActionRLEnvironment`, `Base4ActionEnvironment`, and `Base5ActionEnvironment`, customized for agents selecting from 3, 4, or 5 actions respectively.

`Base3ActionEnvironment` is the simplest, with hold, long, and short options. It supports long-only bots by following the `can_short` flag from the strategy, treating long as entry and short as exit.

`Base4ActionEnvironment` enables entering long, entering short, holding neutral, or exiting any position.

`Base5ActionEnvironment` matches Base4 functionality but separates exit long from exit short actions.

Environment selection primarily affects:
- Available actions in `calculate_reward`
- Actions consumed by the user strategy

All FreqAI environments inherit from `BaseEnvironment`, containing shared logic. The simplest customization involves overriding `calculate_reward()`. More extensive customizations override additional functions within `MyRLEnv` or create entirely new environments inheriting from `BaseEnvironment`.

Only `Base3ActionRLEnv` supports long-only training and trading.

## Live Deployments

FreqAI operates through periodic model retraining cycles in both live and backtesting environments. During live deployment, the system continuously trains new models based on configuration settings and uses the most recent model for predictions on incoming data.

### Automatic Data Download

The platform automatically retrieves the necessary data volume to support model training using parameters like `train_period_days` and `startup_candle_count`.

### Saving Prediction Data

All predictions generated during a model's lifetime are stored in `historic_predictions.pkl`, enabling recovery after system crashes or configuration changes.

### Purging Old Model Data

When running FreqAI for extended periods with frequent retraining, enable `purge_old_models` to automatically delete outdated models while retaining the most recent versions, preserving disk space.

## Backtesting

Backtesting functionality requires pre-downloaded data and involves training separate models for each backtesting window within a specified time range. The system uses a sliding window approach where the training period advances sequentially.

### Saving Backtesting Prediction Data

Predictions generated during backtesting are automatically saved for reuse in subsequent backtests and live deployments using the same model identifier, enabling rapid strategy optimization.

### Backtest Live Collected Predictions

The `--freqai-backtest-live-models` parameter allows reusing predictions generated during dry runs for comparison and analysis purposes.

### Downloading Data for Full Backtest Coverage

For backtesting, data must be downloaded to cover both the training period and the backtesting duration. The required data range should extend backward from the backtest start date by approximately `train_period_days` plus the candle count from `startup_candle_count`.

### Sliding Window Configuration

The backtesting approach divides the timerange into periods defined by `backtest_period_days`, with each period using the preceding `train_period_days` for model training.

## Model Expirations

Model age discrepancies arise during live deployment due to sequential pair training. The `expiration_hours` parameter restricts predictions to models within a specified age threshold.

## Controlling Model Learning

Training parameters are configured through `model_training_parameters` and `data_split_parameters` dictionaries. The `label_period_candles` parameter specifies future offset for training labels.

## Continual Learning

Enabling `continual_learning` allows subsequent training cycles to start from the previous model's final state, providing memory across retraining periods. This feature automatically disables principal component analysis.

## Hyperopt

Strategy hyperoptimization focuses on entry/exit thresholds rather than feature engineering parameters. The system reuses predictions across optimization epochs for efficiency.

## Using Tensorboard

Tensorboard integration (when `activate_tensorboard` is enabled) provides training visualization. Access the interface via command-line execution in a separate terminal session.

# Hyperopt

## Overview

Hyperopt is Freqtrade's hyperparameter optimization tool that uses algorithms from the `optuna` package to discover optimal strategy parameters. The process involves running backtesting repeatedly with different parameter combinations to minimize a loss function.

## Installation

### Docker
Hyperopt dependencies are included in the Docker image.

### Manual Installation
Activate the virtual environment and install the requirements from `requirements-hyperopt.txt`.

## Command Reference

The hyperopt command supports numerous options including:
- `--timeframe`: Specify timeframe (1m, 5m, 30m, 1h, 1d)
- `--timerange`: Define data range for optimization
- `--epochs`: Number of optimization rounds (default: 100)
- `--spaces`: Parameters to optimize (buy, sell, roi, stoploss, trailing, protection, trades)
- `--hyperopt-loss`: Specify loss function for optimization
- `--jobs`: Number of parallel workers
- `--random-state`: Enable reproducible results
- `--enable-protections`: Include protection mechanisms

## Hyperopt Execution Logic

Hyperopt loads historical data and runs `populate_indicators()` once per pair. It then spawns multiple processes that repeatedly execute backtesting with varying parameters. Results pass through the loss function, which guides the algorithm toward better parameter combinations.

## Parameter Definition

Strategy parameters are defined using parameter types:
- **IntParameter**: Integer values within bounds
- **DecimalParameter**: Floating-point values with precision control
- **RealParameter**: Unlimited precision floating-point
- **CategoricalParameter**: Predefined choices
- **BooleanParameter**: True/False options

Parameters must be named with prefixes like `buy_`, `sell_`, `enter_`, `exit_`, or `protection_`, or explicitly assigned via the `space` argument.

## Guards and Triggers

Entry and exit signals should distinguish between:
- **Guards**: Conditions that prevent entry (e.g., "never buy if ADX < 10")
- **Triggers**: Conditions that signal entry (e.g., "buy when EMA5 crosses EMA10")

## Loss Functions

Built-in loss functions include:
- `ShortTradeDurHyperOptLoss`: Default, optimizes trade duration and losses
- `OnlyProfitHyperOptLoss`: Focuses on profit amount
- `SharpeHyperOptLoss`: Optimizes Sharpe Ratio
- `SharpeHyperOptLossDaily`: Sharpe Ratio on daily returns
- `SortinoHyperOptLoss`: Sortino Ratio optimization
- `CalmarHyperOptLoss`: Calmar Ratio optimization
- `MaxDrawDownHyperOptLoss`: Minimizes maximum drawdown
- `ProfitDrawDownHyperOptLoss`: Balances profit and drawdown
- `MultiMetricHyperOptLoss`: Multi-objective optimization

## Executing Hyperopt

### Search Space Optimization

The `--spaces` parameter restricts optimization scope:
- `all`: All parameters including custom spaces
- `default`: All except trailing, trades, protection
- `buy`/`sell`: Entry/exit signals
- `enter`/`exit`: Alternative naming convention
- `roi`: Return on investment table
- `stoploss`: Stop loss value
- `trailing`: Trailing stop parameters
- `protection`: Protection mechanisms
- `trades`: Maximum open trades

## Understanding Results

### ROI Results
Hyperopt generates a ROI table with multiple steps showing profit-taking levels at different timepoints. Results should be copied into the strategy's `minimal_roi` attribute.

### Stoploss Results
The optimized stoploss value replaces the strategy's `stoploss` attribute.

### Trailing Stop Results
Optimized trailing stop parameters include `trailing_stop`, `trailing_stop_positive`, `trailing_stop_positive_offset`, and `trailing_only_offset_is_reached`.

## Parameter Application

When using Hyperoptable parameters, results are automatically exported to a JSON file next to the strategy. Results can also be manually copied into the strategy class as `buy_params`, `sell_params`, and `protection_params` dictionaries.

## Reproducible Results

The first 30 hyperopt epochs are random combinations. Setting `--random-state` to a specific integer value ensures reproducible initial epochs. The random state is logged for future reference.

## Optimizing Protections

Protections can be optimized by defining them as a property returning a list of configurations. Use `space="protection"` parameter designation and run hyperopt with `--spaces protection`.

## Memory Management

To prevent out-of-memory errors:
- Reduce the number of pairs
- Limit timerange with `--timerange`
- Avoid `--timeframe-detail`
- Reduce parallel processes with `-j`
- Use `--analyze-per-epoch` for range-based parameters

## Validation

After optimization, backtest results should match hyperopt results. Verify:
- Parameters are not in `populate_indicators()`
- All optimized values transferred correctly
- Configuration files don't override strategy parameters
- Protections settings match between hyperopt and backtesting
- No parameter JSON files override expected values

# Trading with Leverage

## Multiple Bots on One Account

You cannot run multiple bots on the same account when using leverage. Freqtrade assumes exclusive account access for all leverage trading, as liquidation calculations depend on this assumption.

## Trading with Leverage is Very Risky

Do not use leverage greater than 1x with untested strategies. A strategy must demonstrate positive results in live spot trading first. Consider stoploss levels carefully—with 2x leverage, a 50% stoploss would be dangerously low and could trigger liquidation before the stoploss activates.

**Important:** Only use advanced trading modes when fully understanding how freqtrade and your strategy operate. Never risk more capital than you can afford to lose.

## Shorting

Shorting requires `trading_mode` set to `margin` (currently unavailable) or `futures`, along with `margin_mode` as either `cross` or `isolated`. The strategy class must set `can_short = True` to enable short positions.

## Understanding `trading_mode`

### Spot

The default, low-risk trading mode featuring:
- Long trades only
- No leverage or liquidation
- Profits/losses equal the asset value change minus fees

### Leverage Trading Modes

With leverage, traders borrow capital from the exchange. The capital must be repaid fully, and traders keep profits or pay losses from borrowed capital trades. Exchanges liquidate positions when account value drops sufficiently, ensuring borrowers can repay capital. Liquidation also incurs fees.

**Margin (Currently Unavailable)**

Trading occurs on spot markets with the exchange lending currency equal to chosen leverage. Repayment includes interest, and profits/losses multiply by the leverage factor.

**Futures**

Perpetual swaps are derivative contracts whose prices track underlying assets. Unlike trading actual assets, these contracts can last indefinitely. Traders also exchange funding fees reflecting price differences between contracts and underlying assets.

Configure futures trading with `trading_mode: futures` and `margin_mode: isolated`.

### Pair Namings

Freqtrade follows ccxt naming conventions for futures pairs: `base/quote:settle` (e.g., `ETH/USDT:USDT`).

## Margin Mode

Configure `margin_mode` alongside `trading_mode`. Options are `isolated` or `cross`.

**Isolated Margin Mode**: Each trading pair maintains separate collateral.

**Cross Margin Mode**: A single account shares collateral across trading pairs. Margin draws from total account balance to prevent liquidation. This increases full-account liquidation risk, as losses in one trade affect other trades' liquidation prices.

## Set Leverage to Use

Different strategies require different leverage levels. Freqtrade offers flexibility through strategy leverage callbacks, allowing pair-specific or condition-based leverage adjustments. Without implementation, leverage defaults to 1x.

## Understanding `liquidation_buffer`

Defaults to `0.05`

This ratio creates a safety margin between liquidation and stoploss prices:

`freqtrade_liquidation_price = liquidation_price ± (abs(open_rate - liquidation_price) * liquidation_buffer)`

Use `+` for long trades and `-` for short trades. Acceptable values range from 0.0 to 0.99.

**Example:** A trade at 10 coin/USDT with liquidation at 8 coin/USDT and `liquidation_buffer` of 0.05 produces minimum stoploss of 8.1.

Low or zero `liquidation_buffer` values risk liquidations and fees. Freqtrade calculates liquidation prices but not fees, so liquidations create inaccurate profit/loss reporting. Using `stoploss_on_exchange` is recommended for low buffers when supported.

## Unavailable Funding Rates

Exchanges typically provide futures candles and marks but not always funding rates. Missing data can limit backtesting ranges, producing "No data found" errors. Configure `futures_funding_rate` in settings, ideally set to `0` unless specific rates are known. False rates distort profit calculations in custom exit and stoploss functions.

This approach creates inaccurate backtests but allows testing when exchange data is unavailable.
# Lookahead Analysis

## Overview

Lookahead bias represents a critical challenge in strategy backtesting. This occurs when indicators or signals examine future candles that wouldn't be accessible during live trading. Since backtesting loads entire dataframes into memory simultaneously, strategies can inadvertently "cheat" by referencing data from subsequent periods.

The `lookahead-analysis` command validates strategies by detecting such biases through methodical verification rather than code inspection.

## Command Functionality

The tool chains multiple backtests while systematically testing the strategy to expose lookahead bias. It accomplishes this by comparing indicator values and entry/exit positions between full backtests and segmented backtests, rather than analyzing code directly.

Several configuration parameters are automatically enforced:
- Cache is disabled
- Maximum open trades equals or exceeds pair count
- Starting balance is set to one billion
- Stake amount becomes a fixed 10,000
- Protections remain disabled
- Order types default to market unless `--lookahead-allow-limit-orders` is specified

## Operation Process

Initial execution runs a complete backtest establishing a baseline for indicators and signals. The analysis then proceeds only if the minimum trade threshold is satisfied. Subsequent backtests examine entries and exits individually, comparing results against the baseline to identify any value changes indicating bias.

## Detecting and Addressing Bias

Strategies with substantial lookahead bias typically derive their apparent profitability directly from that bias. Removing problematic conditions or indicators often severely degrades strategy performance. Partial salvage becomes possible only when biased elements represent peripheral components rather than core logic.

## Common Bias Sources

Several patterns frequently introduce lookahead bias:
- "shift(-10)" and similar forward-looking operations
- Direct dataframe row access via `iloc[]`
- Uncontrolled loops accessing future indices
- Aggregation functions like `.mean()` and `.max()` applied across entire datasets without rolling windows
- Technical indicator calculations with inappropriate signal periods

## Results Table Columns

- `filename`: Strategy file identifier
- `strategy`: Class name being analyzed
- `has_bias`: Detection result (Yes/No)
- `total_signals`: Quantity of examined signals
- `biased_entry_signals`: Entry signals containing bias
- `biased_exit_signals`: Exit signals containing bias
- `biased_indicators`: Specific problematic indicator definitions

Note: "biased_exit_signals" may produce false positives when paired with biased entries. Prioritize resolving entry signal bias first.

## Important Limitations

The command only verifies signals that trigger during backtesting. Untriggered signals remain unexamined, potentially yielding false negatives. Limit orders combined with custom price callbacks can produce false positives. FreqAI target indicators flagged as biased in results should be disregarded—these are not actually problematic.

# Plugins

## Pairlists and Pairlist Handlers

Pairlist Handlers define which pairs the bot should trade. They are configured in the `pairlists` section of configuration settings.

The system supports Static Pairlist (using `StaticPairList`) and Dynamic Pairlist options like `VolumePairList`, `CrossMarketPairList`, `MarketCapPairlist`, and `PercentChangePairList`.

- RangeStabilityFilter
- VolatilityFilter

#### Static Pair List

The default method uses pairs from `exchange.pair_whitelist` and respects `exchange.pair_blacklist`. Both support regex patterns like `.*/USDT` for all pairs with USDT stake.

By default, only active pairs are allowed. Setting `"allow_inactive": true` skips validation, useful for backtesting expired pairs.

When positioned after other handlers, `StaticPairList` adds whitelisted pairs to the end.

#### Volume Pair List

`VolumePairList` sorts and filters pairs by trading volume, selecting the top `number_assets` by `quoteVolume` (24-hour quote currency trading volume).

When leading the chain, it ignores `pair_whitelist` and selects from all available markets. When following other handlers, it sorts their outputs.

The `refresh_period` (default 1800 seconds) controls update frequency. Filter instances use current data beyond candle-duration caching.

Minimum and maximum volume thresholds are supported via `min_value` and `max_value`.

**Advanced Mode:** Uses historical candle data to calculate volume over specified timeframes. The `lookback_days` parameter simplifies setup by implying 1-day candles over that period. Alternatively, specify `lookback_timeframe` and `lookback_period` for custom ranges.

When using lookback parameters, `refresh_period` cannot be smaller than candle duration in seconds.

Note: "VolumePairList does not support backtesting mode."

#### Percent Change Pair List

`PercentChangePairList` filters and sorts pairs based on 24-hour price percentage change, helping traders focus on assets with significant movements.

Configuration options include `number_assets`, `min_value`, `max_value`, `sort_direction` (asc/desc), `refresh_period`, and lookback parameters.

The filter operates on previous handler outputs when following others, or selects from all available markets when leading.

Calculation uses: `(Current Close - Previous Close) / Previous Close × 100`.

Note: "PercentChangePairList does not support backtesting mode."

#### ProducerPairList

Reuses pairlist from a Producer without explicit definition per consumer. Requires Consumer mode. The optional `number_assets` parameter limits pairlist length; using 0 or omitting it retains all valid producer pairs.

Can combine with other pairlists and filters, and supports multiple sequential instances from different producers.

#### RemotePairList

Fetches pairlist from remote server or local JSON file, enabling dynamic updates. Configuration includes:
- `mode`: "whitelist" or "blacklist"
- `processing_mode`: "filter" or "append"
- `pairlist_url`: Remote URL or local file path
- `number_assets`: Pair count limit (whitelist mode)
- `refresh_period`: Cache interval
- `keep_pairlist_on_failure`: Use previous list on errors (default true)
- `read_timeout`: Response wait time (default 60 seconds)
- `bearer_token`: Authorization header inclusion
- `save_to_file`: Save processed pairlist to JSON

The remote source must return JSON with pairs and refresh_period fields.

#### MarketCapPairList

Sorts and filters by CoinGecko marketcap rank. In whitelist mode, returns sorted top pairs; in blacklist mode, filters them out.

Configuration includes `number_assets`, `max_rank` (above 250 not recommended due to rate limits), `refresh_period` (default 86,400 seconds), `mode`, and `categories`.

Coins like 1000PEPE/USDT or KPEPE/USDT are detected via 1000 and K prefixes.

#### CrossMarketPairList

Generates or filters pairs based availability across market types. The `pairs_exist_on` setting determines whether pairs must exist on both spot and futures (`both_markets`) or only the current market (`current_market_only`).

#### AgeFilter

Removes pairs listed less than `min_days_listed` days (default 10) or more than `max_days_listed` days (default unlimited). New listings experience volatility during price discovery, so this filter prevents early entries.

#### DelistFilter

Removes pairs delisting within `max_days_from_now` days (default 0 removes all future delistings). Available on Bybit Futures, Bitget Futures, and Binance. Binance Futures supports dry and live modes; Binance Spot limited to live mode.

Note: "DelistFilter does not support backtesting mode."

#### FullTradesFilter

Shrinks whitelist to in-trade pairs when slots are full (`max_open_trades` isn't -1). Prevents unnecessary indicator calculations and reduces CPU usage when no new trades can open.

Recommended placement: directly below the primary pairlist.

Note: "FullTradesFilter does not support backtesting mode."

#### OffsetFilter

Offsets incoming pairlist by `offset` value. Useful for removing top X pairs or splitting pairlists across bot instances.

Example: `offset: 10` with `number_assets: 20` takes items 10-30 from the initial list.

Warning: Offsets larger than pairlist length produce empty results.

#### PerformanceFilter

Sorts pairs by past trade performance:
1. Positive performance
2. No closed trades yet
3. Negative performance

Trade count acts as tiebreaker. The optional `minutes` parameter considers rolling-window performance; omitting or setting to 0 uses all-time data.

The `min_profit` parameter (as ratio, 0.01 = 1%) filters pairs below threshold. Using this without `minutes` risks empty pairlists.

Requires startup period with 100+ trades in database.

Note: "PerformanceFilter does not support backtesting mode."

#### PrecisionFilter

Filters low-value coins preventing proper stoploss placement. Pairs are excluded if 1% variance in stop price causes rounding precision issues that would trigger `rounded(stop_price) <= rounded(stop_price * 0.99)`.

Note: "PrecisionFilter does not support backtesting mode using multiple strategies."

#### PriceFilter

Filters by price using:
- `min_price`: Removes pairs below specified price
- `max_price`: Removes pairs above specified price
- `max_value`: Removes pairs with excessive minimum value change
- `low_price_ratio`: Removes pairs where 1 pip exceeds ratio threshold

At least one setting must apply.

#### ShuffleFilter

Randomizes pair order. Default shuffles once per candle; set `shuffle_frequency` to "iteration" for each iteration instead.

Optional `seed` value enables reproducible results for backtesting (auto-detected).

#### SpreadFilter

Removes pairs with bid-ask spread exceeding `max_spread_ratio` (default 0.005).

Calculation: `1 - bid/ask`. Example: bid 0.00000026 and ask 0.00000027 yields ratio ~0.037, exceeding 0.005 threshold.

#### RangeStabilityFilter

Removes pairs where lowest low to highest high difference over `lookback_days` falls below `min_rate_of_change` or exceeds `max_rate_of_change`. Results cache for `refresh_period`.

Useful for removing stable coins with minimal trading range and extreme volatility pairs.

Optional `sort_direction` ("asc" or "desc") enables sorting.

#### VolatilityFilter

Removes pairs with average volatility outside bounds. Volatility measures historical variation using standard deviation of logarithmic daily returns. In normal distribution, 68% observations fall within one standard deviation.

Configuration uses `lookback_days`, `min_volatility`, `max_volatility`, and `refresh_period`.

Optional `sort_direction` ("asc" or "desc") enables sorting.

### Full Example of Pairlist Handlers

A complete configuration chain demonstrates practical usage: starting with `VolumePairList` for top 20 assets, applying `DelistFilter` and `AgeFilter` to remove delisting and new pairs, using `PrecisionFilter` and `PriceFilter` for tradeable pairs, applying `SpreadFilter` and `VolatilityFilter` for quality pairs, and finally shuffling with fixed seed for reproducibility.

## Protections

Protections prevent unexpected events by temporarily halting trades for specific pairs or all pairs. Lock times round to the next candle.

### Available Protections

- `StoplossGuard`: Stops trading after stoploss threshold
- `MaxDrawdown`: Halts trading when max-drawdown exceeded
- `LowProfitPairs`: Locks low-profit pairs
- `CooldownPeriod`: Prevents re-entry immediately after exit

### Common Settings to All Protections

| Parameter | Description |
|-----------|-------------|
| `method` | Protection name from available options |
| `stop_duration_candles` | Lock duration in candles |
| `stop_duration` | Lock duration in minutes (not usable with candles variant) |
| `lookback_period_candles` | Recent trade window in candles |
| `lookback_period` | Recent trade window in minutes (incompatible with candles variant) |
| `trade_limit` | Minimum trade count threshold |
| `unlock_at` | Regular unlock time in HH:MM format |

Durations can be specified in minutes or candles.

#### Stoploss Guard

Evaluates trades within `lookback_period`. If `trade_limit` or more trades hit stoploss, trading stops for `stop_duration`.

Applies across all pairs unless `only_per_pair: true` limits to individual pairs.

For futures, `only_per_side: true` considers long and short separately, allowing one side continuation.

`required_profit` sets minimum acceptable profit ratio (defaults to 0.0 for all losses).

Considers `"stop_loss"`, `"stoploss_on_exchange"`, and `"trailing_stop_loss"` results with negative profit.

#### MaxDrawdown

Evaluates trades closed within `lookback_period`. Supports two calculation modes:

- `ratios` (default): Legacy approximation from cumulative profit ratios
- `equity`: Standard peak-to-trough drawdown on account equity using starting balance and cumulative absolute profit

If drawdown exceeds `max_allowed_drawdown`, trading stops for `stop_duration` after the last trade.

New setups should use equity mode; ratios mode is for backward compatibility.

#### Low Profit Pairs

Uses trades within `lookback_period` to calculate pair profit ratio. If below `required_profit`, that pair locks for `stop_duration`.

For futures, `only_per_side: true` locks individual sides.

#### Cooldown Period

Locks pair for `stop_duration` after closing trade, preventing immediate re-entry. Applies only at pair level, never globally. Ignores `lookback_period`.

### Full Example of Protections

A comprehensive example using 1-hour timeframe demonstrates layered protection:
- `CooldownPeriod`: 5-candle pair cooldown post-exit
- `MaxDrawdown`: Stop 4 candles if >20% drawdown with 20 trades over 48 candles
- `StoplossGuard`: Stop 2 candles after 4 stoploss events within 24 candles
- `LowProfitPairs`: Lock 60 minutes if <2% profit with 2 trades in 6 candles
- `LowProfitPairs`: Lock 2 candles if <1% profit with 4 trades in 24 candles

Multiple instances with different parameters create increasing barriers for underperforming pairs. Evaluation occurs in defined sequence.
# Producer/Consumer Mode

## Overview

Freqtrade enables a "producer/consumer" architecture where a downstream instance (consumer) listens to messages from an upstream instance (producer) via message websocket. This mechanism allows "reuse of computed indicators (and signals) for pairs in multiple bots without needing to compute them multiple times."

The producer broadcasts `analyzed_df` and `whitelist` messages, eliminating redundant indicator calculations across multiple bot instances.

## Configuration

Enable consumer mode by adding an `external_message_consumer` section to the configuration file with these required parameters:

- **enabled**: Boolean flag to activate consumer mode (defaults to false)
- **producers**: Array of producer configurations, each requiring:
  - name: Identifier for the producer instance
  - host: Hostname or IP address of the producer
  - port: Port number (defaults to 8080)
  - ws_token: Authentication token matching producer's configuration
  - secure: Optional SSL/TLS flag (defaults to false)

Optional settings include `wait_timeout`, `ping_timeout`, `sleep_time`, `remove_entry_exit_signals`, `initial_candle_limit`, and `message_size_limit`.

## Implementation Pattern

Producer instances require no special strategy modifications—they calculate indicators normally in `populate_indicators()` and broadcast results.

Consumer strategies must:
- Set `process_only_new_candles = False`
- Use `self.dp.get_producer_df(pair)` to retrieve analyzed dataframes
- Merge producer dataframes with local data using `merge_informative_pair()`
- Reference producer columns with appropriate suffixes in entry/exit logic

This architecture supports distributed trading across resource-constrained devices while maintaining centralized indicator calculation on more powerful machines.
# Recursive Analysis

This documentation explains how to validate strategies for inaccuracies caused by recursive indicator issues in Freqtrade.

## Overview

A recursive formula defines sequence terms relative to preceding term(s). This matters for Freqtrade because backtesting uses full historical data, while dry/live runs are limited by exchange API data availability.

For example, an indicator calculated using the latest 1000 candles produces different values than one calculated using 500 candles, creating potential discrepancies between backtest and live trading results.

## Purpose

The `recursive-analysis` command prepares different data lengths and calculates indicators based on them. It compares indicator values across varying `startup_candle_count` values to measure variance from a baseline calculation.

## Command Settings

- Use `-p` to specify a pair (preferably high-price, moderate volatility pairs like BTC or ETH)
- Set a long timerange of at least 5000 candles for accurate benchmark calculations
- `--cache` is forced to "none" to prevent automatic loading of previous calculations

The command also performs a basic lookahead bias check on indicator values only.

## Why Odd-Numbered Default Startup Candles?

When exchanges return candle data via API (e.g., Binance's 1000-candle limit), the last candle is current while preceding candles are "startup candles."

Setting startup count to 1000 instead of 999 forces pagination—the API returns two groups (1000 + 1 candles), resulting in 1999 startup candles instead of 999. This compensates for exchange API pagination behavior.

## How the Command Works

1. Initial indicator calculation using supplied timerange generates a benchmark
2. Additional runs execute for each different startup candle count value
3. Last-row indicator values across all startup counts are compared
4. Differences are reported in a variance table

## Understanding Output

The results table shows variance percentages compared to the benchmark:

- `nan%` indicates insufficient data to calculate that indicator
- `-` represents zero variance (exact match to benchmark)
- Percentage values show divergence from baseline

Users should assess whether variance is small enough that indicators won't affect trading decisions.

## Caveats

- Analysis only compares last-row indicator values; actual trading impact isn't assessed
- Goal is sufficiently low variance, not necessarily zero variance
- Indicators like EMA use recursive formulas, so some variance is expected
- Only `populate_indicators` and `@informative` decorators are analyzed; calculations in `populate_entry_trend` or `populate_exit_trend` are ignored

# Stop Loss

## Overview

The `stoploss` configuration parameter defines a loss ratio that triggers a sale. A value of `-0.10` sells immediately if profit drops below -10% for a trade. Stoploss calculations include fees, positioning the stop exactly 10% below the entry point.

"All stoploss properties mentioned in this file can be set in the Strategy, or in the configuration. Configuration values will override the strategy values."

## Stop Loss On-Exchange/Freqtrade

Stoploss modes can operate either on-exchange or off-exchange using these configuration values:

- `emergency_exit`: defaults to 'market'
- `stoploss_on_exchange`: Boolean flag
- `stoploss_on_exchange_interval`: 60 seconds (default)
- `stoploss_on_exchange_limit_ratio`: 0.99 (default)

### Supported Exchanges

On-exchange stoploss is available for Binance, Bingx, Bitmart, Bitget, Bybit, Gate.io, HTX, Hyperliquid, Kraken, OKX, Bitvavo, and Kucoin, with varying support for spot and futures trading across market and limit order types.

### stoploss_on_exchange and stoploss_on_exchange_limit_ratio

Enabling stop loss on exchange places a stoploss limit order immediately after a buy fills, protecting against sudden crashes through in-exchange execution without network overhead. For limit orders, two prices are needed: the stoploss price and a limit price slightly below it.

Example: "Calculation example: we bought the asset at 100$. Stop-price is 95$, then limit would be `95 * 0.99 = 94.05$`."

If cancelled manually on the exchange, the bot will create a new stoploss order.

### stoploss_on_exchange_interval

This parameter configures how frequently (in seconds) the bot checks and updates the stoploss. The default 60-second interval prevents exchange bans from excessive updates. This logic also reapplies stoploss orders if accidentally cancelled.

### stoploss_price_type

For futures markets only, this setting determines which price type triggers stoploss: "last", "mark", or "index". Exchange support varies. Freqtrade validates this setting on startup and ignores it for spot markets.

### force_exit

An optional value defaulting to the same as `exit`, used when sending a `/forceexit` command via Telegram or REST API.

### force_entry

An optional value defaulting to the same as `entry`, used when sending a `/forceentry` command via Telegram or REST API.

### emergency_exit

Defaults to "market" and activates when stop loss on exchange order placement fails.

## Stop Loss Types

### Static Stop Loss

A simple approach where you define a fixed loss ratio. "For example, simplified math: the bot buys an asset at a price of 100$. The stop loss is defined at -10%. The stop loss would get triggered once the asset drops below 90$."

### Trailing Stop Loss

Automatically moves the stop loss upward as price increases. "In summary: The stoploss will be adjusted to be always be -10% of the highest observed price."

Enable with:
- `stoploss = -0.10`
- `trailing_stop = True`

### Trailing Stop Loss with Different Positive Loss

Uses a standard stoploss until profitability reaches an offset, then switches to a different trailing value. "Use an offset to change your stoploss. Use `trailing_stop_positive_offset` to ensure that your new trailing stoploss will be in profit."

Configuration:
- `trailing_stop = True`
- `trailing_stop_positive`: new stoploss percentage
- `trailing_stop_positive_offset`: profit threshold

### Trailing Stop Loss Only Once Offset Reached

Keeps stoploss static until reaching a specified offset, then trails to lock profits. Set `trailing_only_offset_is_reached = True` to activate only after the offset triggers.

## Stoploss and Leverage

Stoploss represents risk on a trade—how much you're willing to lose. "When using leverage, the same principle is applied - with stoploss defining the risk on the trade (the amount you are willing to lose)."

A 10% stoploss at 10x leverage triggers on a 1% price move. Exercise caution with tight stoploss values under leverage, as trades need room to breathe.

## Changing Stoploss on Open Trades

Modify stoploss values via configuration or strategy changes using `/reload_config`, or restart the bot. New values apply to open trades with corresponding log entries.

### Limitations

"Stoploss values cannot be changed if `trailing_stop` is enabled and the stoploss has already been adjusted."

# Strategy Migration between V2 and V3

To support new markets and trade-types (short trades and leverage trading), the Freqtrade interface has undergone significant changes. While compatibility with existing spot market strategies is maintained, migration is necessary for accessing new functionality.

## Quick Summary / Migration Checklist

### Strategy Methods

- `populate_buy_trend()` → `populate_entry_trend()`
- `populate_sell_trend()` → `populate_exit_trend()`
- `custom_sell()` → `custom_exit()`
- `check_buy_timeout()` → `check_entry_timeout()`
- `check_sell_timeout()` → `check_exit_timeout()`

New `side` argument (either "long" or "short") added to:
- `custom_stake_amount`
- `confirm_trade_entry`
- `custom_entry_price`

### Dataframe Columns

- `buy` → `enter_long`
- `sell` → `exit_long`
- `buy_tag` → `enter_tag`
- New columns: `enter_short` and `exit_short`

### Trade Object Changes

New properties:
- `is_short`
- `entry_side`
- `exit_side`
- `trade_direction`
- Renamed: `sell_reason` → `exit_reason`

Renamed: `trade.nr_of_successful_buys` → `trade.nr_of_successful_entries`

### Configuration Settings

- `INTERFACE_VERSION` should be set to 3
- `order_time_in_force`: "buy" → "entry", "sell" → "exit"
- `order_types`: buy → entry, sell → exit
- `unfilledtimeout`: buy → entry, sell → exit
- Strategy-level settings: `use_sell_signal` → `use_exit_signal`, `sell_profit_only` → `exit_profit_only`, etc.

### Terminology Updates

Order pricing configuration renamed:
- `bid_strategy` → `entry_pricing`
- `ask_strategy` → `exit_pricing`
- `ask_last_balance` and `bid_last_balance` → `price_last_balance`

Webhook terminology updated from "sell"/"buy" to "exit"/"entry"

Telegram notifications updated similarly

## Extensive Explanation

### populate_entry_trend

Change method name and column assignments from `buy` to `enter_long`. This function identifies long entry signals using technical indicators and dataframe conditions.

### populate_exit_trend

Rename `populate_sell_trend()` to `populate_exit_trend()` and change column from `sell` to `exit_long`. Implement separate `enter_short` and `exit_short` columns for short trading signals.

### custom_exit

The renamed `custom_exit()` method now executes for every iteration regardless of profit settings, allowing more flexible exit logic based on current conditions.

### Timeout Methods

Both `check_entry_timeout()` and `check_exit_timeout()` now receive an `Order` object instead of a dictionary, providing improved type safety and structure.

### custom_stake_amount

Accept new `side` parameter indicating trade direction. This enables different stake sizing strategies for long versus short positions.

### confirm_trade_entry

Add `side` parameter to distinguish between long and short entry confirmations. Adjust validation logic accordingly.

### confirm_trade_exit

Replace `sell_reason` parameter with `exit_reason` for consistency with new terminology. The old parameter remains briefly for compatibility.

### custom_entry_price

Now receives optional `Trade` object and requires `side` parameter. This allows price customization based on trade direction and existing position data.

### Adjust Trade Position

Use `trade.nr_of_successful_entries` instead of `trade.nr_of_successful_buys` to accurately count all entry types.

### Helper Methods

The `stoploss_from_open()` and `stoploss_from_absolute()` functions now require `is_short` parameter (from `trade.is_short`). The latter also accepts optional `leverage` parameter.

### Strategy/Configuration Settings

#### order_time_in_force

Update dictionary keys from "buy"/"sell" to "entry"/"exit" with uppercase values.

#### order_types

Rename all order type keys following the entry/exit convention, including `emergency_exit`, `force_exit`, and `force_entry`.

#### Strategy Level Settings

Replace these attribute names:
- `use_sell_signal` → `use_exit_signal`
- `sell_profit_only` → `exit_profit_only`
- `sell_profit_offset` → `exit_profit_offset`
- `ignore_roi_if_buy_signal` → `ignore_roi_if_entry_signal`

#### unfilledtimeout

Update dictionary structure to use `entry` and `exit` keys.

#### Order Pricing

Restructure pricing configuration with new names and optional `same`/`other` price-side options. The `ignore_buying_expired_candle_after` setting remains supported.

## FreqAI Strategy Migration

The `populate_any_indicators()` method splits into four specialized functions:

- `feature_engineering_expand_all()` - automatically expanded features
- `feature_engineering_expand_basic()` - basic features without period expansion
- `feature_engineering_standard()` - final custom feature extraction
- `set_freqai_targets()` - dedicated target definition

### Feature Engineering Functions

Remove manual expansion loops and pair/timeframe prefixes. The framework now handles automatic duplication across configured periods and timeframes. Prefix features with `%` for recognition.

### set_freqai_targets

Define model targets in this dedicated function using `&` prefix. Targets specify what the model should predict based on feature inputs.

### Data Pipeline Updates

Custom `IFreqaiModel` implementations should replace `data_cleaning_train/predict()` methods with `define_data_pipeline()` and `define_label_pipeline()` functions. These provide unified data normalization and transformation throughout training and prediction phases.
# Freqtrade Strategies 101: A Quick Start for Strategy Development

## Required Knowledge

A strategy in Freqtrade is a Python class defining the logic for buying and selling cryptocurrency assets. Assets are represented as pairs combining a coin with a stake currency.

Data comes from exchanges as candles containing six values: date, open, high, low, close, and volume. Technical analysis functions process candle data to produce indicators, which are analyzed to generate signals. Signals become orders on cryptocurrency exchanges, resulting in trades.

Freqtrade distinguishes between entry and exit rather than buy and sell, supporting both long and short positions:

- **Long trades**: Buy a coin using a stake currency, profit when the coin value increases
- **Short trades**: Borrow capital from the exchange, repay later at a lower rate to profit from price decreases

## Structure of a Basic Strategy

### Main Dataframe

Strategies use tabular dataframes with rows and columns to generate signals. Each pair maintains its own dataframe indexed by date. The five primary columns represent OHLCV data (open, high, low, close, volume).

### Populate Indicator Values

The `populate_indicators` function adds columns representing technical analysis indicator values. Common indicators include Relative Strength Index, Bollinger Bands, Money Flow Index, Moving Average, and Average True Range. Columns are added by calling technical analysis functions and assigning results to dataframe columns.

### Populate Entry Signals

The `populate_entry_trend` function defines entry conditions. When the `enter_long` column contains a value of 1, Freqtrade recognizes an entry signal. For short trades, use the `enter_short` column instead.

### Populate Exit Signals

The `populate_exit_trend` function defines exit conditions. A value of 1 in the `exit_long` column signals an exit. For short trades, use `exit_short`.

## A Simple Strategy

A minimal strategy requires three main components: indicator population, entry trend definition, and exit trend definition. The strategy specifies a timeframe, stoploss percentage, and minimal ROI targets.

## Making Trades

When entry or exit signals appear (a 1 in relevant columns), Freqtrade attempts to create trade orders. Each new position occupies a slot, with the maximum concurrent open trades defined by the `max_open_trades` configuration option.

Several scenarios prevent signal execution:

- Insufficient stake remaining to buy or insufficient funds to sell
- No available trade slots
- Existing open trade for that pair
- Colliding entry and exit signals on the same candle
- Strategy rejection via entry or exit callbacks

## Backtesting and Forward Testing

Two testing modes are available:

**Backtesting** uses historical exchange data for rapid performance assessment. However, results can be distorted to appear more profitable than reality.

**Dry run (forward testing)** uses real-time exchange data without executing actual trades. Results take longer but provide more reliable performance indicators than backtesting.

Dry runs are enabled by setting `dry_run` to true in configuration.

### Assessing Results

Always compare backtesting and dry run results. Expected differences include:

- Backtesting assumes all orders fill; dry runs may see unfilled limit orders
- Entry timing differs due to signal processing delays
- Profit calculations vary with entry rate differences
- Computational lag affects price unpredictability

## Controlling or Monitoring a Running Bot

Six mechanisms manage running bots:

- **FreqUI**: Web interface for monitoring and control
- **Telegram**: Mobile alerts and limited control
- **FTUI**: Terminal interface for monitoring only
- **freqtrade-client**: Python REST API implementation
- **REST API endpoints**: Programmatic bot interaction
- **Webhooks**: External service integration (Discord, etc.)

### Logs

Freqtrade generates extensive debugging logs to standard output. The `--logfile` option writes logs to files instead.

## Final Thoughts

Algorithmic trading is challenging. Most public strategies underperform due to the effort required for consistent profitability across multiple scenarios. Backtests alone are unreliable for performance assessment.

No single configuration fixes a poorly performing strategy. Freqtrade offers tools for due diligence, and the community provides support through Discord. Always invest only what you can afford to lose.

## Conclusion

Strategy development involves defining entry and exit signals based on technical indicators. Additional guidance is available in the strategy customization documentation and FAQ.
# Advanced Strategy

This documentation covers advanced concepts for Freqtrade strategies. Familiarity with basic Freqtrade concepts and strategy customization is recommended before exploring these topics.

## Storing Information (Persistent)

Freqtrade enables storage and retrieval of custom user data associated with specific trades in the database. Information is stored using `trade.set_custom_data(key='my_key', value=my_value)` and retrieved using `trade.get_custom_data(key='my_key')`.

Each data entry connects a trade with a user-supplied string key. This feature is available only in callbacks that provide a trade object. Data must be JSON-serializable for database storage, with automatic conversion on retrieval.

Simple data types (bool, int, float, str) are recommended to ensure proper serialization. Large data chunks may cause unintended side effects, including database growth and slowdowns.

The Trade object provides three accessors:

- `trade.get_custom_data(key='something', default=0)` returns the actual value
- `trade.get_custom_data_entry(key='something')` returns the entry with metadata accessible via `.value`
- `trade.set_custom_data(key='something', value={'some': 'value'})` sets or updates the key

## Storing Information (Non-Persistent)

Non-persistent storage through custom dictionaries is deprecated. Custom dictionaries prefixed with `custom_` can store pair-specific data, but this data does not persist after bot restart or config reload.

**Warning:** Data volume should remain small to prevent memory consumption issues. Always use pair as a dictionary key for pair-specific data.

## Dataframe Access

Dataframe queries are available in various strategy functions using the dataprovider. The `get_analyzed_dataframe()` method returns only candles accessible to backtesting.

Important considerations include:

- Use `.iloc[-1]` only with `get_analyzed_dataframe()` results, not in `populate_*` methods
- Trade open dates must be rounded to match candle dates
- Query results may be empty for newly opened trades

## Enter Tag

Multiple entry signals can be named and accessed in exit logic. Tags allow tracking which specific signal triggered an entry.

Entry tags are limited to 255 characters. When multiple signals collide or conditions deactivate signals, the last written tag persists ("last write wins" behavior).

## Exit Tag

Similar to entry tagging, exit signals can be specified and labeled. The provided exit-tag becomes the exit-reason displayed in backtest results.

Exit reasons are limited to 100 characters.

## Strategy Version

Custom strategy versioning is available through implementing a `version()` method returning the desired version string.

Proper version control (such as Git) should accompany this method, as Freqtrade does not maintain historic strategy versions.

## Derived Strategies

Strategies can inherit from other strategies, avoiding code duplication. Child strategies override specific attributes and methods while maintaining inherited functionality.

Separate strategy files are recommended to prevent issues with hyperopt parameter files, using imports for parent strategy classes.

## Embedding Strategies

Strategies can be embedded directly in configuration files using BASE64 encoding. The encoded string is placed in the strategy configuration field as `"NameOfStrategy:BASE64String"`.

Strategy names must match exactly between the class definition and configuration reference.

## Performance Warning

Pandas may issue fragmentation warnings during strategy execution. This occurs when repeatedly adding DataFrame columns in loops. The solution involves using `pd.concat(axis=1)` to combine multiple DataFrames efficiently.

Freqtrade automatically runs `dataframe.copy()` after `populate_indicators()`, minimizing performance implications of fragmentation.
# Strategy Callbacks

## Overview

Strategy callbacks in Freqtrade are functions called "whenever needed" during bot operation, as opposed to main strategy functions that execute once during backtesting. These callbacks enable dynamic decision-making based on current trade conditions.

Key principle: "avoid doing heavy calculations in callbacks to avoid delays during operations."

## Available Callbacks

The following callbacks are currently supported:

- `bot_start()` - Called once when strategy loads
- `bot_loop_start()` - Called at start of each bot iteration
- `custom_stake_amount()` - Manages position sizing before trade entry
- `custom_exit()` - Defines custom exit signals
- `custom_stoploss()` - Implements dynamic stoploss logic
- `custom_roi()` - Sets custom return-on-investment thresholds
- `custom_entry_price()` and `custom_exit_price()` - Customizes order pricing
- `check_entry_timeout()` and `check_exit_timeout()` - Handles unfilled order timeouts
- `confirm_trade_entry()` and `confirm_trade_exit()` - Final order confirmation before placement
- `adjust_trade_position()` - Enables DCA and position adjustments
- `adjust_entry_price()` and `adjust_exit_price()` - Refreshes limit orders
- `leverage()` - Sets leverage for futures trading
- `order_filled()` - Executes actions after orders fill
- `plot_annotations()` - Adds chart annotations for visualization

## Exit Logic Comparison

| Function | Best For | Avoid For |
|----------|----------|-----------|
| `populate_exit_trend()` | Vectorized indicator-based exits | Per-trade customization; trade-specific data |
| `custom_exit()` | Individual trade conditions; profit-based decisions | Vectorized indicator logic; stoploss replacement |
| `custom_stoploss()` | Dynamic trailing stops; trade-based conditions | Immediate exits; static stoploss |
| `custom_roi()` | Minimum profit thresholds; take-profit logic | Immediate exits; static ROI |

## Core Callbacks Explained

### Bot Start and Loop Start

`bot_start()` executes once after strategy instantiation, useful for loading remote data or initializing resources. `bot_loop_start()` runs at each throttling iteration (approximately every 5 seconds in live/dry mode, or once per candle during backtesting).

### Stake Size Management

The `custom_stake_amount()` callback allows dynamic position sizing based on current conditions. Returning zero or None prevents trade entry. The bot clamps returned values to the valid range and logs this action.

### Custom Exit Signals

`custom_exit()` enables condition-specific trade closure for individual positions. Returning a non-empty string or True signals an exit, with strings limited to 64 characters. This method ignores `exit_profit_only` settings.

### Custom Stoploss

The `custom_stoploss()` callback returns a percentage relative to current price. Key behaviors:

- Stoploss can only move upward; lower values are ignored
- The initial `stoploss` parameter serves as an absolute minimum
- Returns None to indicate no change desired
- The `after_fill` parameter enables stoploss adjustment after position adjustments
- Time-based calculations should use `current_time`, not `datetime.now()`

### Custom ROI

`custom_roi()` specifies minimum profit thresholds. If both `minimal_roi` and `custom_roi` are defined, the lower threshold triggers exit. Return None to fall back to `minimal_roi` logic.

### Custom Order Pricing

`custom_entry_price()` and `custom_exit_price()` customize order prices before placement. Invalid or missing returns default to `proposed_rate`. Maximum distance from current price is configurable (default 2%).

### Order Timeout Rules

`check_entry_timeout()` and `check_exit_timeout()` determine whether unfilled orders should be cancelled. Return True to cancel, False to maintain the order. These callbacks are useful for price-dependent timeout strategies.

### Trade Entry and Exit Confirmation

`confirm_trade_entry()` and `confirm_trade_exit()` provide final order approval before exchange placement. Returning False aborts the operation. Exit confirmation may be called multiple times for different exit reasons in sequence: exit_signal, stop_loss, roi, trailing_stop_loss.

### Position Adjustment

The `adjust_trade_position()` callback enables DCA (Dollar Cost Averaging) and position management. Positive values increase position; negative values decrease it. Enable with `position_adjustment_enable = True`. This feature:

- Allows returns of tuples with adjustment amount and tag
- Is called frequently, requiring performance optimization
- Ignores `max_open_trades` limits for additional orders
- Calls frequently during backtesting (every candle)

Partial exits use the formula: "amount = negative_stake × trade.amount ÷ trade.stake_amount"

### Leverage Callback

The `leverage()` callback sets leverage for futures markets (default 1.0 for no leverage). Values exceeding `max_leverage` are automatically adjusted.

### Order Filled Callback

`order_filled()` executes after any order completion (entry, exit, stoploss, or position adjustment). Useful for storing candle data or updating trade metadata.

### Plot Annotations

`plot_annotations()` returns annotation objects for chart display during freqUI requests. Supported types include areas, lines, and points with customizable styling and positioning.

---

**Important Notes:**

- All time-based calculations should reference `current_time` parameter, not system time
- Trailing stop should be disabled when using custom stoploss
- Loose logic in `adjust_trade_position()` can cause repeated entries every 5 seconds
- Position adjustments with high frequency may impact memory and performance
- `confirm_trade_exit()` should not block stoploss exits to avoid significant losses
# Strategy Customization

This page explains how to customize strategies, add new indicators, and set up trading rules for Freqtrade.

## Develop Your Own Strategy

### Generating a Strategy Template

Use the command `freqtrade new-strategy --strategy AwesomeStrategy` to create a new strategy. This generates a file at `user_data/strategies/AwesomeStrategy.py`. The strategy name differs from the filename—Freqtrade uses the strategy name in most commands.

Template levels can be controlled with the `--template` parameter: use `minimal` for an empty strategy or `advanced` for more complex features.

### Anatomy of a Strategy

A strategy file contains:

- Candle data in OHLCV format
- Indicators
- Entry logic with signals
- Exit logic with signals, minimal ROI, and callbacks
- Stoploss (fixed, trailing, or with callbacks)
- Optional pricing and position adjustment

The bot includes a sample called `SampleStrategy` in `user_data/strategies/sample_strategy.py`. Set `INTERFACE_VERSION` to define the strategy interface version (current version is 3).

### Bot Modes

Freqtrade processes strategies in five modes: backtesting, hyperopting, dry mode, live mode, and FreqAI. Always test in dry mode before live trading.

## Diving Deeper

### Dataframe

Freqtrade uses pandas DataFrames to store candlestick (OHLCV) data. Each row represents one candle, with the latest complete candle last.

Operations must use vectorization across the entire dataframe rather than simple Python comparisons.

#### Why Can't I See Real-Time Candle Data?

Freqtrade only stores complete candles. Incomplete/unfinished candles aren't available in the dataframe to prevent "repainting."

### Customize Indicators

Add indicators by extending `populate_indicators()`. Only include indicators used in entry/exit logic or populating other indicators to maintain performance.

Always return the dataframe without modifying mandatory columns: "open," "high," "low," "close," "volume."

#### Indicator Libraries

Out of the box, Freqtrade installs: ta-lib, pandas-ta, and technical. Additional libraries can be installed as needed.

### Strategy Startup Period

Some indicators require an unstable startup period with insufficient candle data (NaN values). Set the `startup_candle_count` attribute to the maximum candles needed for stable indicator calculation.

When using informative pairs with higher timeframes, `startup_candle_count` reflects the maximum period any informative timeframe needs.

Use recursive-analysis to verify the correct value—when variance shows 0%, you have sufficient startup data.

**Note:** `startup_candle_count` should remain below `ohlcv_candle_limit * 5` (typically 2,500 for most exchanges).

### Entry Signal Rules

Edit `populate_entry_trend()` to define entry logic. This method must create an `"enter_long"` column (mandatory, even for short-only strategies) with values of 1 for entries and 0 for no action.

Use `"enter_tag"` to name signals for later debugging. For short entries, create `"enter_short"` signals. Ensure `can_short = True` is set if shorting.

### Exit Signal Rules

Edit `populate_exit_trend()` to define exit logic. Create `"exit_long"` (or `"exit_short"` for shorts) columns with 1 for exits and 0 for no action.

Use `"exit_tag"` to name exit signals. The `use_exit_signal` configuration can suppress exit signals without affecting entry collision rules.

### Minimal ROI

The `minimal_roi` variable defines minimum Return On Investment before exiting, independent of exit signals. It's a dictionary with time in minutes as keys and profit percentage as values.

To disable ROI, set `minimal_roi = {}`.

Calculations based on candle duration can use timeframe conversion to adjust times dynamically.

### Stoploss

Setting a stoploss is recommended to protect capital. Example: `stoploss = -0.10` for a 10% stoploss. See the dedicated stoploss documentation for full features.

### Timeframe

This is the candle periodicity the bot uses. Common values include "1m," "5m," "15m," and "1h." Access this within strategy methods as `self.timeframe`.

### Can Short

Set `can_short = True` to enable short signals in futures markets. Strategies with this setting fail on spot markets.

### Metadata Dict

The `metadata` dictionary (available in `populate_*` methods) contains pair information accessible via `metadata['pair']`, returning formats like "XRP/BTC" or "XRP/BTC:BTC" for futures.

Don't modify the metadata dict; use the Storing Information section for persistent data instead.

## Imports Necessary for a Strategy

Recommended base imports include NumPy, pandas, datetime utilities, and Freqtrade-specific modules like IStrategy, Trade, Order, PairLocks, and technical indicator libraries.

## Strategy File Loading

Freqtrade loads strategies from `.py` files in the user directory (default: `user_data/strategies`). Start the bot by passing the class name to `--strategy`. Check available strategies with `freqtrade list-strategies`.

Customize the strategy directory using `--strategy-path user_data/otherPath`.

## Informative Pairs

### Get Data for Non-Tradeable Pairs

Informative pairs provide data on wider timeframes for reference. OHLCV data downloads automatically during whitelist refresh.

Specify pairs as tuples: `("pair", "timeframe")`. These pairs won't trade unless in the whitelist or selected by dynamic whitelisting.

### Informative Pairs Decorator (@informative())

Use the `@informative()` decorator on `populate_indicators_*` methods to easily define informative pairs. All decorated methods run in isolation with merged dataframes passed to main `populate_indicators()`.

When hyperopting, use `.range` instead of `.value` attributes.

### merge_informative_pair()

This method safely merges informative pairs without lookahead bias, renaming columns uniquely and using forward-fill to maintain values throughout periods.

Using informative timeframes smaller than the main timeframe isn't recommended.

## Additional Data (DataProvider)

Access the DataProvider for additional data in your strategy. It provides:

- `available_pairs` – Cached pairs with timeframes
- `current_whitelist()` – Current whitelisted pairs
- `get_pair_dataframe(pair, timeframe)` – Historical or cached data
- `get_analyzed_dataframe(pair, timeframe)` – Analyzed dataframe after indicators
- `orderbook(pair, maximum)` – Current order book
- `ticker(pair)` – Current ticker data
- `check_delisting(pair)` – Delisting schedule if any
- `funding_rate(pair)` – Current funding rate for futures
- `send_msg()` – Send custom notifications

DataProvider is available during hyperopt only within `populate_indicators()`, not in entry/exit methods.

### Available Pairs

Iterate through cached pairs with `for pair, timeframe in self.dp.available_pairs:`

### current_whitelist()

Returns the current whitelist, useful for dynamic whitelists like VolumePairlist. Not supported for plot-dataframe or FreqUI webserver mode.

### get_pair_dataframe(pair, timeframe)

Fetches live or historical OHLCV data. In backtesting, avoid looking into the future within `populate_*()` methods.

### get_analyzed_dataframe(pair, timeframe)

Returns the analyzed dataframe with signals and the latest analysis time. Returns empty dataframe if pair wasn't cached.

### orderbook(pair, maximum)

Retrieves current order book structured as bids/asks arrays. Returns None in case of failure. Unavailable during backtesting.

### ticker(pair)

Returns current ticker data. Values vary by exchange (some don't return VWAP or last). Always verify data and add error handling. Not available during backtesting.

### check_delisting(pair)

Returns delisting datetime if scheduled, otherwise None. Limited exchange availability. Returns up-to-date values; check runmode during backtesting.

### funding_rate(pair)

For futures pairs in format "base/quote:settle," returns current funding rate structure with mark price, funding rate, and timestamps. Vary by exchange availability.

### Send Notification

Use `self.dp.send_msg()` to send custom notifications. Identical notifications send once per candle unless `always_send=True`. Only works in live/dry-run modes.

## Additional Data (Wallets)

Access wallet information via the `wallets` object. Always check availability to avoid backtesting failures.

Methods: `get_free(asset)`, `get_used(asset)`, `get_total(asset)`.

## Additional Data (Trades)

Query trade history from the database using `Trade.get_trades_proxy()`. Not available in `populate_*()` methods during backtesting/hyperopt.

## Prevent Trades for Specific Pairs

Freqtrade automatically locks pairs for the current candle after exit to prevent immediate re-entry.

### Locking Pairs from Within the Strategy

Use `self.lock_pair(pair, until, [reason])` where `until` is a future datetime. Unlock with `self.unlock_pair(pair)` or `self.unlock_reason(reason)`. Verify lock status with `self.is_pair_locked(pair)`.

Locks round up to the next candle. Manually locking pairs isn't available during backtesting.

## Print the Main Dataframe

Insert print statements in `populate_entry_trend()` or `populate_exit_trend()` to inspect dataframe contents.

## Common Mistakes When Developing Strategies

### Looking Into the Future While Backtesting

Backtesting analyzes the entire timerange at once, requiring care to avoid using future data. Avoid:

- `shift(-1)` or negative shifts
- `.iloc[-1]` or absolute positioning in `populate_*()` functions
- Functions using all dataframe values (use rolling calculations instead)
- `.resample('1h')` without the right border label
- Plain `.merge()` for longer timeframes (use informative pair helpers)

Use lookahead-analysis and recursive-analysis tools to identify these issues.

### Colliding Signals

When entry and exit signals conflict (both set to 1), Freqtrade ignores the entry to prevent immediate exit. Entry signals are ignored if multiple conflicting signals exist:

- `enter_long` conflicts with `exit_long` and `enter_short`
- `enter_short` conflicts with `exit_short` and `enter_long`

## Further Strategy Ideas

Visit the strategy repository for examples. Use these for learning, always backtest before dry-running and live trading.

After developing a strategy, learn how to use backtesting to evaluate performance.

## Setup your Telegram bot

### 1. Create your Telegram bot

Start a conversation with the Telegram BotFather and send `/newbot`. Follow the prompts to choose a public name and username for your bot (usernames must end in "bot").

BotFather will return an API token in the format `22222222:APITOKEN`. Save this token for the config parameter `token`.

Remember to start a conversation with your bot by clicking the `/START` button.

### 2. Telegram user_id

#### Get your user id

Contact the userinfobot on Telegram to retrieve your user ID. Use this value for the config parameter `chat_id`.

Add the bot to a group and issue the `/tg_info` command to retrieve the group ID without needing external tools. The response includes the group ID in a format ready for configuration.

For Freqtrade configuration, use the complete value as a string: `"chat_id": "-1001332619709"`

**Note on group usage:** All group members gain access to your bot and its commands. Only add trusted individuals to avoid security issues.

##### Group Topic ID

Use the `topic_id` parameter to direct the bot to respond in a specific group topic. Without this setting, responses go to the general channel if topics are enabled.

#### Authorized users

Limit command access in groups by specifying user IDs in `"authorized_users"`. An empty list prevents all users from controlling the bot; non-authorized users receive messages but cannot issue commands.

## Control telegram noise

Adjust message verbosity using three settings: `on` (sends message with notification), `silent` (sends message without sound), and `off` (skips message entirely).

Configure notification types including status updates, warnings, startup messages, entry/exit signals, and protection triggers. The `*_fill` notifications default to off and require explicit enabling. Use `balance_dust_level` to define minimum balance thresholds for display.

Define custom command buttons in your config. The default layout provides three rows with commonly used commands.

## Telegram commands in action

### /start

Returns: **Status:** `running`

### /pause | /stopentry | /stopbuy

Returns message indicating paused state. The bot stops opening new trades but continues managing existing positions. Position adjustments remain active on the exit side only. Run `/start` to resume trading.

**Important:** The pause signal only persists while running and resets upon restart.

### /stop

Returns: `Stopping trader ...` **Status:** `stopped`

### /status

For each open trade, displays trade ID, pair, direction, leverage, amount, entry tag, open rate, current rate, unrealized profit, and stoploss level.

### /status table

Returns open trades in table format with ID, direction, pair, duration, and profit percentage.

### /count

Shows current trade count versus maximum available trades.

### /profit

Displays ROI from closed trades, total trade count, bot start date, first/latest trade times, average duration, best performing pair, trading volume, profit factor, win/loss ratio, winrate, expectancy ratio, and maximum drawdown.

The relative profit represents average profit per trade, while sigma profit reflects returns based on starting capital.

### /forceexit

Confirms exit with exchange, pair, rate, and estimated profit. Calling without parameters shows a list of open trades to exit.

### /forcelong [rate] | /forceshort [rate]

Confirms entry with exchange, pair, limit rate, and trade amount. Omitting the pair triggers a selection prompt. Requires `force_entry_enable` configuration.

### /performance

Lists top-performing pairs with profit amounts and percentages, calculated against total investment per currency.

### /balance

Shows cryptocurrency balance with available, total, and pending amounts for each currency.

### /daily

Displays daily profit over the last 7 days by default. Includes day, trade count, profit in base currency, profit in USD, and percentage change.

### /weekly

Shows weekly profit over the last 8 weeks by default, with weeks starting Monday.

### /monthly

Displays monthly profit over the last 6 months by default.

### /whitelist

Shows the current trading pair whitelist.

### /blacklist [pair]

Displays current blacklist or adds new pairs. Use `/reload_config` to reset.

### /version

Returns the bot's version number.

### /marketdir

Updates or displays the current market direction setting. The value persists only during the current session and resets on restart. Available directions: `long`, `short`, `even`, or `none`.

Access this value in strategies via `self.market_direction`. Note that this feature is intended for manual live/dry trading and produces unreliable backtesting results due to its non-persisted nature.
# Trade Object

## Trade

A position entered by freqtrade is stored in a `Trade` object, which persists to the database. This represents a core concept in freqtrade and appears throughout the documentation. The object is passed to strategies in many strategy callbacks, though it cannot be directly modified. Indirect modifications may result from callback operations.

## Trade - Available attributes

The following attributes/properties are available for individual trades using `trade.<property>` syntax (e.g., `trade.pair`):

| Attribute | DataType | Description |
|-----------|----------|-------------|
| `pair` | string | Pair of this trade |
| `safe_base_currency` | string | Compatibility layer for base currency |
| `safe_quote_currency` | string | Compatibility layer for quote currency |
| `is_open` | boolean | Whether the trade is currently open or concluded |
| `exchange` | string | Exchange where trade was executed |
| `open_rate` | float | Entry rate (average if trade-adjusted) |
| `open_rate_requested` | float | Requested rate when trade opened |

## Class methods

Class methods return generic information and typically query the database. Use them as `Trade.<method>()`.

**Note:** Most methods work in both backtesting/hyperopt and live/dry modes. During backtesting, usage is limited to strategy callbacks. Use in `populate_*()` methods is unsupported and produces incorrect results.

### get_trades_proxy

When strategies need information on existing trades, use `Trade.get_trades_proxy()`.

Supported keyword arguments (all optional):
- `pair` (e.g., `pair='ETH/USDT'`)
- `is_open` (e.g., `is_open=False`)
- `open_date` (e.g., `open_date=current_date - timedelta(days=2)`)
- `close_date` (e.g., `close_date=current_date - timedelta(days=5)`)

Calling without arguments returns all trades in the database.

### get_open_trade_count

Returns the number of currently open trades.

### get_total_closed_profit

Retrieves total profit generated, aggregating the absolute profit values across all closed trades.

### total_open_trades_stakes

Retrieves the total stake amount currently in active trades.

## Class methods not supported in backtesting/hyperopt

### get_overall_performance

Retrieves overall performance similar to the `/performance` telegram command. Returns a list with pair, profit ratio, and trade count. Only available in live or dry-run modes.

### get_trading_volume

Gets total trading volume based on orders.

# Updating Freqtrade

## How to Update

To update your freqtrade installation, use one of the methods below that corresponds to your installation approach.

### Tracking Changes

Breaking changes and modified behavior are documented in the changelog accompanying each release. For the develop branch, monitor pull requests to stay informed about upcoming changes.

## Why Update?

Maintaining current software ensures you benefit from the latest features and improvements. Regular updates are essential for smooth operation, as "Freqtrade depends heavily on underlying exchange APIs, which change frequently across different exchanges." Consistent updates preserve compatibility.

## Docker

For legacy installations using the `master` image, migrate to the stable release image by updating your docker file and replacing `freqtradeorg/freqtrade:master` with `freqtradeorg/freqtrade:stable`.

Run `docker compose pull` followed by `docker compose up -d` to fetch and start the updated image.

## Installation via Setup Script

Execute the setup script with the `--update` flag.

**Note:** Run this command with your virtual environment disabled.

## Plain Native Installation

Update dependencies to prevent unexpected issues:

Update the repository, install dependencies, and ensure freqUI is current.

## Problems Updating

Update difficulties typically stem from missing dependencies or failed dependency installations. While major platforms have wheel availability for heavy dependencies, sometimes installation is not possible.

Refer to common installation problem sections for guidance.
# Utility Sub-commands

## Overview

The Freqtrade bot includes several utility subcommands beyond live trading, dry-run, backtesting, hyperopt, and data downloading functions. These tools assist with configuration, strategy management, and analysis.

## Create userdir

This command builds the directory structure for Freqtrade files. It generates sample strategy and hyperopt examples for getting started. The `--reset` flag restores sample files to their default state, though this may overwrite existing data without confirmation.

The resulting directory structure includes folders for backtest results, historical data, hyperopt results, custom hyperopt loss functions, Jupyter notebooks, plotting outputs, and strategies.

## Create new config

This utility generates a new configuration file by prompting users about essential settings. Only fundamental questions are asked; additional configuration options exist in the comprehensive Configuration documentation.

The interactive setup asks about dry-run enablement, stake currency, stake amount, maximum open trades, timeframe, display currency, exchange selection, and Telegram integration preferences.

## Show config

This command displays the configuration file with sensitive values redacted by default. It proves especially helpful when using split configuration files or environment variables, as it shows the merged result.

A warning notes that while known sensitive information is removed automatically, users should verify the output to prevent accidental exposure of private data.

## Create new strategy

This generates a new strategy from a template without overwriting existing files. The filename matches the class name and gets placed in `user_data/strategies/`.

Three template options exist: minimal, full (with multiple sample indicators), and advanced (containing all optional functions and methods).

## List Strategies

The `list-strategies` subcommand displays all strategies in a specific directory. Strategies with errors appear in red as "LOAD FAILED," while duplicate-named strategies display in yellow as "DUPLICATE NAME."

A security warning notes that loading strategies executes all module-level Python code, presenting risks if untrusted files exist in the directory.

## List Hyperopt-Loss functions

The `list-hyperoptloss` subcommand shows all available hyperopt loss functions. Like the strategies command, it flags problematic modules in red for failed loads and yellow for duplicate names.

## List freqAI models

Use `list-freqaimodels` to view available FreqAI models. It applies the same color-coding system for identifying load failures and duplicate names.

## List Exchanges

The `list-exchanges` subcommand presents exchanges available to the bot. The `-a` flag shows all exchanges supported by the ccxt library, including those known to have compatibility issues. Trading mode can be specified as spot, margin, or futures.

## List Timeframes

This command displays timeframes available for a specified exchange, helping users understand what candle intervals are supported for their chosen market.

## List pairs/list markets

These subcommands enumerate pairs and markets on an exchange. Pairs feature a "/" separating base and quote currencies. The stake currency defines which quote currency Freqtrade uses. Filtering options include `--base` and `--quote` parameters.

By default, only active pairs display. The `-a` flag shows all pairs, including inactive ones. Pairs may appear as untradeable if the smallest tradeable price falls below a threshold.

## Test pairlist

The `test-pairlist` subcommand validates dynamic pairlist configurations. It can generate static pairlists for backtesting and hyperopt use.

## Convert database

The `convert-db` command migrates databases between systems (such as sqlite to postgres), transferring all trades, orders, and pairlocks. Users must ensure the target database is empty before conversion.

## Webserver mode

This experimental mode enhances backtesting and strategy development by keeping data loaded between runs when timeframe and timerange remain constant. FreqUI controls backtesting processes and displays results.

For Docker deployments, the webserver can be started with port configuration or by modifying the docker-compose command.

## Show previous Backtest results

The `backtesting-show` command displays previous backtest results. The `--show-pair-list` option outputs a sorted list of pairs for easy copy-pasting into configurations.

A caution warns that using only winning pairs may lead to strategy overfitting, which won't perform well on future data.

## Detailed backtest analysis

The `backtesting-analysis` command provides advanced analysis of backtest results. It supports grouping by entry tags, exit tags, pairs, and other metrics. Users can filter by specific signals and export results to CSV format.

## List Hyperopt results

The `hyperopt-list` subcommand shows previously evaluated hyperopt epochs. Multiple filtering options exist for profits, trade counts, average times, and objectives. Results can export to CSV or display as JSON.

## Show details of Hyperopt results

The `hyperopt-show` command displays specific epoch details. Users can select by index, show only best epochs, or filter profitable ones. Results support JSON output and optional breakdown by time periods.

## Show trades

The `show-trades` command prints selected or all trades from the database. Users can specify individual trade IDs and request JSON formatting.

## Strategy-Updater

This utility converts strategies to v3 compliance on a best-effort basis. Original strategies are preserved in `user_data/strategies_orig_updater/`. Users should verify conversion results and apply Python formatting tools afterward.
# Webhook Usage

## Configuration

Enable webhooks by adding a webhook section to your configuration file and setting `webhook.enabled` to `true`.

A sample configuration uses IFTTT with fields like `value1`, `value2`, and `value3` for entry, entry_cancel, entry_fill, exit, exit_cancel, exit_fill, and status events. The webhook URL should point to your webhook service endpoint.

You can set the POST body format to Form-Encoded (default), JSON-Encoded, or raw data using `"format": "form"`, `"format": "json"`, or `"format": "raw"` respectively.

For Form-Encoded or JSON-Encoded configurations, you can configure any number of payload values. With raw data format, you can only configure one value named `"data"`, and it will not be output in the POST request—only its value.

### Nested Webhook Configuration

Some webhook targets require nested structures, which can be accomplished by setting content as a dictionary or list instead of text directly. This is only supported for JSON format.

## Additional Configurations

The `webhook.retries` parameter sets the maximum number of retries if the webhook request is unsuccessful (defaults to `0`). The `webhook.retry_delay` parameter specifies time in seconds between retries (defaults to `0.1`).

You can also specify `webhook.timeout`, which defines how long the bot waits before assuming the host is unresponsive (defaults to 10 seconds).

Custom messages can be sent via the `self.dp.send_msg()` function from within strategies by setting `allow_custom_messages` to `true`.

## Webhook Message Types

### Entry / Entry Fill

Fields in `webhook.entry` and `webhook.entry_fill` are filled when the bot places or fills an order to increase a position. Parameters include: trade_id, exchange, pair, direction, leverage, open_rate, amount, open_date, stake_amount, stake_currency, base_currency, quote_currency, fiat_currency, order_type, current_rate, and enter_tag.

### Entry Cancel

Fields in `webhook.entry_cancel` are filled when the bot cancels a long/short order. Available parameters match entry-related fields.

### Exit / Exit Fill

Fields in `webhook.exit` and `webhook.exit_fill` are filled when the bot places or fills an exit order. Parameters include: trade_id, exchange, pair, direction, leverage, gain, amount, open_rate, close_rate, current_rate, profit_amount, profit_ratio, stake_currency, base_currency, quote_currency, fiat_currency, enter_tag, exit_reason, order_type, open_date, close_date, sub_trade, and is_final_exit.

### Exit Cancel

Fields in `webhook.exit_cancel` are filled when the bot cancels an exit order. Available parameters match exit-related fields.

### Status

Fields in `webhook.status` are used for regular status messages. The only possible parameter is `{status}`.

## Discord

A special webhook form is available for Discord. Configuration includes webhook_url and arrays for exit_fill and entry_fill events, with exit_fill and entry_fill being optional and defaulting to predefined configurations.

Custom messages can be sent from strategies to Discord via the dataprovider.send_msg() function by enabling the `allow_custom_messages` option.
